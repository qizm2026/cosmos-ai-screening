决策树不在纸上，而是在代码逻辑里。对话引擎（app/api/chat/route.ts）中有一个四分支的判断流程，每次 AI 回复后，系统解析 AI
  的"内部小纸条"（SYS 块），然后走不同的路。

  用大白话描述：

  学生回复了一条消息
    │
    ├─ 先检测：有没有风险信号？（自杀/自伤关键词）
    │   ├─ 第一次命中 → AI 用自然对话确认，不弹窗
    │   ├─ 第二次命中 → 自动确认高风险 → 触发 Q9 兜底
    │   └─ 没有 → 继续
    │
    └─ AI 的回复里有没有 SYS 块？
        │
        ├─ 有 SYS 块
        │   ├─ info_sufficiency >= 2（说得够清楚）
        │   │   → 推进到下一个话题
        │   │   → 存入 AI 随手记的分数
        │   │
        │   ├─ info_sufficiency <= 1 + 追问次数还没用完
        │   │   → 继续追问，告诉 AI 该追问什么方向
        │   │
        │   ├─ info_sufficiency = 1 + 追问次数用完了
        │   │   → forced-choice（二选一，不兜底）
        │   │
        │   └─ info_sufficiency = 0 + 追问次数用完了
        │       → 硬兜底（展示 PHQ-9 原题 + 四个选项）
        │
        └─ 没有 SYS 块（AI 忘记了）
            ├─ 第一轮 → 等待 AI 追问，不干预
            └─ 已经追问过 → 代码自己判断学生说的够不够清楚
                ├─ 够清楚 → 推进
                └─ 不够清楚 → 硬兜底

  核心参数只有一个：每个话题最多追问 2 次。2 次之后还说不清楚，就走兜底。
   ---
  二、AgentMental 论文的决策树是怎么实现的？

  AgentMental 比 COSMO 复杂得多。它用了4 个独立的 AI 智能体，各司其职：

  AGq（提问者）    → 专门负责生成问题
  AGev（评估者）   → 专门负责判断"学生说清楚了吗"（输出 0-2 分）
  AGs（评分者）    → 专门负责给当前话题打分
  AGu（更新者）    → 专门负责维护记忆 + 跨话题修正分数

  它的决策逻辑是数学公式驱动的，不是代码分支：

  AGev 给信息充分性打分（0/1/2）
    → 分数 > θ（阈值，论文设为 1）且 追问次数 < d（上限，论文设为 3）
        → AGq 生成针对性追问
    → 否则
        → 推进到下一个话题

  关键差异：

  ┌──────────┬─────────────────────────┬───────────────────────────────────────┐
  │          │       AgentMental       │                 COSMO                 │
  ├──────────┼─────────────────────────┼───────────────────────────────────────┤
  │ 谁做判断 │ 独立的 AGev 智能体      │ 同一个 AI 在 SYS 块中自己判断         │
  ├──────────┼─────────────────────────┼───────────────────────────────────────┤
  │ 判断精度 │ 0/1/2 三级              │ 0/1/2 三级（V4.0 升级后）             │
  ├──────────┼─────────────────────────┼───────────────────────────────────────┤
  │ 追问上限 │ d=3                     │ MAX_ITEM_ROUNDS=2                     │
  ├──────────┼─────────────────────────┼───────────────────────────────────────┤
  │ 追问方向 │ AGq 按五维度定向追问    │ AI 自由发挥 + missing_dimensions 提示 │
  ├──────────┼─────────────────────────┼───────────────────────────────────────┤
  │ 兜底     │ forced-choice（二选一） │ forced-choice + 硬兜底（四选一）      │
  ├──────────┼─────────────────────────┼───────────────────────────────────────┤
  │ 架构     │ 4 个独立 Agent          │ 单 AI + 代码层决策                    │
  └──────────┴─────────────────────────┴───────────────────────────────────────┘

---
  三、记忆结构：COSMO 现在是怎么实现的？

  COSMO 的记忆是平面结构——几个独立的数组和字典，互不关联。

  具体存了什么：

  1. messages（聊天记录）：完整的对话历史，一问一答全记着
  2. asked_questions（问过的问题）：AI 每次提的问题存下来，避免重复问
  3. user_context（用户画像）：用正则表达式从学生的话里抓取——职业、年龄、爱好、提到的症状
  4. item_summaries（话题摘要）：每个聊完的话题，AI 写一段 2-3 句的小结（含情绪、频率、持续时间、症状、影响五个维度）
  5. item_scores_initial（初评分）：每个话题 AI 随手记的分数

  这些数据之间没有关联结构。就像一本笔记本，每页独立记不同的东西，没索引。

  四、AgentMental 论文的记忆结构是怎么实现的？

  AgentMental 用的是三层树形结构，数据之间有明确的关系：

  根节点：用户基本信息（职业、性别、年龄）
    │
    ├── 话题节点1：兴趣减退
    │   ├── 分数：1
    │   ├── 行为摘要："学生偶尔对活动失去兴趣..."
    │   └── 陈述节点（每轮追问一个）：
    │       ├── 情绪：挫败感
    │       ├── 频率：每周 2-3 次
    │       ├── 持续时间：约两个月
    │       ├── 症状：不想画画，不追剧
    │       └── 影响：社交减少
    │
    ├── 话题节点2：情绪低落
    │   ├── 分数：2
    │   ├── 行为摘要："..."
    │   └── 陈述节点：...
    │
    └── ...（其余话题）

  每次聊完一个话题后，更新智能体（AGu）会做两件事：
  1. 把这个话题的五维信息整理好，挂到对应的话题节点下
  2. 回头检查之前所有话题的摘要——有没有新信息暗示之前的摘要需要补充修正

---
  COSMO 的三个引擎 ≠ AgentMental 的四个 Agent
     
  这是两种完全不同的"拆分"逻辑。

  COSMO 的拆分：按时间阶段拆分

  三个引擎是串行的——一个做完才轮到下一个：

  对话引擎 ──────────────→ 评分引擎 ──────────────→ 报告引擎
  （聊天过程中运行）        （聊天结束后运行）        （评分完成后运行）

  - 对话引擎：负责"怎么聊天"——问问题、听回答、决定追问还是推进
  - 评分引擎：负责"怎么打分"——聊天全部结束后，读完整记录，给 9 个话题打分
  - 报告引擎：负责"怎么呈现"——把分数变成学生能看懂的文字

  这三个引擎之间不互相通信、不互相校验。对话引擎给了初评分，评分引擎会参考但不是必须认。报告引擎拿到评分结果就直接用。

  AgentMental 的拆分：按功能角色拆分

  四个 Agent 是并行的——在对话过程中同时工作、互相通信：

  AGq（提问）↔ AGev（评估）↔ AGs（评分）↔ AGu（记忆+校准）
      ↑            ↑            ↑            ↑
      每轮都在     每轮都在      每轮都在      每轮都在

  - AGq 负责生成问题
  - AGev 负责判断"学生说清楚了吗"
  - AGs 负责给当前话题打分（不等聊天结束，当场就打）
  - AGu 负责更新记忆、根据新信息修正前面话题的分数

  关键区别：AgentMental 的评分（AGs）和校准（AGu）是嵌在对话过程中的——每聊完一个话题就当场打分、当场校准。COSMO
  的评分是对话全部结束后才做一次。

  ---
  为什么 COSMO 不拆成多个 Agent？

  两个原因，一个技术、一个务实。

  技术原因：AgentMental 用 4 个 Agent 跑在 AutoGen 框架上，每个 Agent 都是独立的 LLM 调用。4 个 Agent × 9 个话题 ×
  每话题最多 3 轮追问 = 上百次 LLM 调用。COSMO 用 DeepSeek API 按 token 计费，这么跑成本会非常高。

  务实原因：COSMO 把 AgentMental 用独立 Agent 做的事情，交给了同一个 LLM 的内部推理。比如 AGev 的"评估信息充分性"，COSMO
  是让 AI 在 SYS 块里写 info_sufficiency: 0/1/2——本质上 LLM
  在生成回复的同时，在自己的"脑子"里做了评估。这比额外调用一个评估 Agent 快得多也便宜得多。

  ├──────────────┼───────────────────────────────┼──────────────────────────────────┤
  │ 校准时机     │ 评分时一次性校准              │ 每完成一个话题就回头校准前面所有 │

  └──────────────┴───────────────────────────────┴──────────────────────────────────┘

  ---
  本质差异

  COSMO 的拆分逻辑是**"把一件事分成三个阶段做"**——有点像流水线。

  AgentMental 的拆分逻辑是**"同一件事分给四个不同专长的人同时做"**——像手术室里主刀、麻醉师、护士各司其职。

  COSMO 的选择是务实的——用流水线架构 + SYS 块（让一个 AI 同时干多个角色的活）换来了更低的延迟和成本。代价是缺少 AgentMental
  那种"每聊完一个话题就回头校准"的渐进式精度。
