## 一、优化概览

### 1.1 核心设计决策：两阶段评分（对标 AgentMental）

```
对话中（逐项初评 → AGs 角色）              对话后（全局校准 → AGu 角色）
SYS 块 item_score_guess                  /api/score PsyCoT + 跨条目校验
        ↓                                          ↓
session.item_scores_initial             session.score_result（可覆盖初评）
```

**为什么？** 逐项初评时 LLM 只能看到当前条目对话，看不到后续条目信息——比如 Q1 打了 2 分但 Q6 的回答暗示 Q1 实际没那么重。全局校准阶段 LLM 看到全部 9 条目的完整对话，可以修正初评偏差。这正是 AgentMental 论文 `[S', R] = AGu(H, M_topic)` 的设计思路。

## 三、Phase 1：两阶段评分引擎（对标 AgentMental AGs + AGu）

### 3.0 设计思路

对标 AgentMental 的两阶段评分架构：

```
AgentMental:                              COSMO V4.1:
--------------------                      --------------------
AGs（逐项初评）                            SYS 块 item_score_guess
  ↓                                         ↓
AGu（全局校准 + inter-topic correlation）   /api/score PsyCoT + 跨条目交叉校验
  ↓                                         ↓
S'（最终分数集）                           session.score_result（可覆盖初评）
```

**第一阶段（对话中逐项初评）**：LLM 在 SYS 块中输出 `item_score_guess`（0-3/-1），基于当前条目的对话内容即时打分。这个初评分写入 `session.item_scores_initial`。

**第二阶段（对话后全局校准）**：`/api/score` 进行 PsyCoT 三步推理，LLM 看到 Q1~Q9 全部对话后逐项复核。对于初评分与全局判断不一致的条目，LLM 有权在 `matched_anchor_index` 中覆盖——初评分仅作为参考输入，不是最终输出。

**为什么需要两阶段？** 对话中逐项初评存在信息不对称——后续条目的回答可能改变对前面条目的判断。全局校准阶段 LLM 看到完整上下文后可以修正偏差。

### 3.3 PsyCoT 两阶段评分 Prompt

**文件：** `src/lib/prompts/score-prompt.ts`

**改造：** 替换 `buildScoreSystemPrompt` 中的「评分逻辑」段落。

**新文案：**

```
【评分逻辑（两阶段：逐项复核 + 全局校准，对标 AgentMental AGs→AGu）】

第一阶段 — 逐项复核（AGs 角色）
对每个条目逐一进行 PsyCoT 三步推理：

Step 1 — 症状锚定
从对话中提取该条目相关的具体事实，引用学生原话中的关键词句。
- 示例："学生提到'最近两周基本每天都睡不好，躺下去一个小时才能睡着'"
- 如果对话中完全没有该维度的有效信息 → 标记"症状信息缺失"，
  在 matched_anchor_index 填 0，reason 写"对话信息不足，默认 0 分"，跳过后续步骤

Step 2 — 严重度判断
综合以下维度判断症状严重程度：
- 频率：多少天出现一次？持续多久了？
- 强度：轻度/中度/重度？
- 影响范围：有没有影响到学习、社交、日常起居？
将判断结果用一两句话写入 severity_judgment 字段。

Step 3 — 锚点匹配
将 Step 2 的严重度判断与下方评分锚点对照，输出最匹配的锚点索引（0/1/2/3）。
- matched_anchor_index：从 0-3 中选择最匹配的锚点编号
- reason：说明为什么选这个锚点，综合 Step 1 的证据和 Step 2 的判断

第二阶段 — 全局校准（AGu 角色，对标 AgentMental inter-topic correlation）
逐项复核完成后，通读全部 9 个条目的评分结果，做交叉校验：
- 检查是否存在矛盾：某个条目初评偏高，但其他条目的回答暗示该维度没那么严重？
  例如：Q1（兴趣缺失）初评 2 分，但 Q2-Q9 的对话显示学生仍保持正常社交和学习——是否应该下调 Q1？
- 检查是否存在遗漏：某个条目初评偏低，但多条目对话中的交叉线索暗示低估？
- 综合校准：将需要修正的条目调整 matched_anchor_index，在 reason 中说明修正原因。
  - 示例 reason："全局校准：初评 Q1=2，但 Q2-Q8 显示学生社交功能正常、无明显兴趣丧失表现，下调至 1"
- 如果初评分与全局判断一致 → 不需要调整，reason 中无需特别说明。
```

## 四、Phase 2：追问引擎优化

### 4.1 SysAnalysis 接口升级

**文件：** `app/api/chat/route.ts`

```typescript
// 现状
interface SysAnalysis {
  item_score_guess: number
  info_sufficient: boolean
  risk_confirmed: boolean
}

// V4.0
interface SysAnalysis {
  item_score_guess: number
  info_sufficiency: number          // 0=严重不足 / 1=部分线索 / 2=信息充分
  risk_confirmed: boolean
  missing_dimensions?: string[]     // 可选，如 ["frequency", "impact", "duration"]
  item_summary?: {                  // 可选，Phase 3 使用
    summary: string
    emotion: string
    frequency: string
    duration: string
    symptom: string
    impact: string
  }
}
```

### 4.2 parseSysBlock() 兼容逻辑

```typescript
function parseSysBlock(text: string): SysAnalysis | null {
  const match = text.match(/<!--SYS\n([\s\S]*?)\nSYS-->/)
  if (!match) return null
  try {
    const parsed = JSON.parse(match[1])
    
    // 向后兼容：旧格式 info_sufficient (boolean) → 映射为 info_sufficiency (number)
    let info_sufficiency: number
    if (typeof parsed.info_sufficiency === 'number') {
      info_sufficiency = Math.max(0, Math.min(2, Math.round(parsed.info_sufficiency)))
    } else if (typeof parsed.info_sufficient === 'boolean') {
      info_sufficiency = parsed.info_sufficient ? 2 : 0
    } else {
      info_sufficiency = 0  // 默认兜底
    }

    return {
      item_score_guess: typeof parsed.item_score_guess === 'number' ? parsed.item_score_guess : -1,
      info_sufficiency,
      risk_confirmed: parsed.risk_confirmed === true,
      missing_dimensions: Array.isArray(parsed.missing_dimensions) ? parsed.missing_dimensions : undefined,
      item_summary: parsed.item_summary || undefined,
    }
  } catch {
    return null
  }
}
```

### 4.3 决策树重构

**文件：** `app/api/chat/route.ts`，覆盖推进逻辑段落

**替换现有 if-else 链：**

```typescript
if (session.phase === 'interview' && item) {
  if (shouldTriggerQ9) {
    // 路径 A：高风险确认 → Q9 兜底（与现状一致）

      newCurrentItemIndex = Math.min(session.current_item_index + 1, 8)
      newItemRounds = 0

      // Phase 3：收集条目摘要

        newItemSummaries = collectItemSummary(session, item.id, sys.item_summary)
      }
    }
    else if (rounds < MAX_ITEM_ROUNDS) {
      // 信息不足 + 还有追问额度 → 追问
      newItemRounds = rounds + 1
      newMissingDimensions = sys.missing_dimensions ?? []
    }
    else {
      // 信息不足 + 追问额度用尽
      if (suff === 1) {
        // 部分信息 → forced-choice（LLM 二选一，本轮不兜底）
        newItemRounds = rounds + 1  // 进入 rounds=2 或 3
        // 无 immediate fallback，LLM 被 prompt 指示进行二选一
      } else {
        // suff === 0 → 硬兜底
        show_fallback = true
        fallback_item = item.id
        fallback_options = [...item.fallback_options_original]
        sessionReply = item.fallback_prompt
        updatedCoverage[item.id as ItemId] = 'fallback'
        newItemAnswerQuality[item.id as ItemId] = 'insufficient'
        newItemRounds = 0
        newFollowUpCount = 0
        newCurrentConfidenceScore = 0
      }
    }
  }
  else {
    // ===== SYS 缺失 =====
    if (newItemRounds === 0) {
      // 第一轮，等 AI 追问
      newItemRounds = 1
    } else {
      // 已追问，代码层兜底
      const vagueness = detectAnswerVagueness(userMessage)
      newCurrentConfidenceScore = vagueness.confidenceScore
      if (vagueness.isVague && vagueness.confidenceScore <= 1) {
        // 代码层判定信息不足 → 硬兜底
        show_fallback = true
        fallback_item = item.id
        fallback_options = [...item.fallback_options_original]
        sessionReply = item.fallback_prompt
        updatedCoverage[item.id as ItemId] = 'fallback'
        newItemAnswerQuality[item.id as ItemId] = 'insufficient'
        newItemRounds = 0
        newFollowUpCount = 0
      } else {
        // 代码层判定信息可接受 → 正常推进
        updatedCoverage[item.id as ItemId] = 'answered'
        newItemAnswerQuality[item.id as ItemId] = vagueness.confidenceScore >= 4 ? 'sufficient' : 'partial'
        newCurrentItemIndex = Math.min(session.current_item_index + 1, 8)
        newItemRounds = 0
        newFollowUpCount = 0
        lastCoverageUpdate = { item: item.id as ItemId, status: 'answered' }
      }
    }
  }
  
  // forced-choice 后仍无法确定：item_rounds >= MAX_ITEM_ROUNDS + 1 且用户仍模糊
  if (!show_fallback && newItemRounds > MAX_ITEM_ROUNDS) {
    const vagueness = detectAnswerVagueness(userMessage)
    if (vagueness.isVague && vagueness.confidenceScore <= 1) {
      show_fallback = true
      fallback_item = item.id
      fallback_options = [...item.fallback_options_original]
      sessionReply = item.fallback_prompt
      updatedCoverage[item.id as ItemId] = 'fallback'
      newItemAnswerQuality[item.id as ItemId] = 'insufficient'
      newItemRounds = 0
    }
  }
}
```

### 4.4 Prompt 更新

**文件：** `src/lib/prompts/chat-prompt.ts`

**SYS 块格式文档更新：**

将现有的 `info_sufficient` 说明替换为：

```
- info_sufficiency: 信息充分度评分
  0 = 严重不足：学生说的和话题完全对不上，或者只说了"还行""不知道"
  1 = 部分线索：有一些信息但不足以判断严重程度，还缺一些关键维度
  2 = 信息充分：有足够的信息来推测得分

- missing_dimensions: （可选）缺失的信息维度。
  从 [频率, 持续时间, 具体表现, 影响程度, 触发情境] 中选择
  只有当 info_sufficiency <= 1 时才需要填写
```

**新增 forced-choice 指令：**

在 `confidenceHint` 逻辑之后新增：


## 六、实施顺序与验证

### 执行顺序

1. **P0**（纯 prompt 改动 + 删除代码）
2. **P1**（两阶段评分，依赖 P2 的 SYS 接口升级——`missing_dimensions` 和 `item_summary` 字段需先在 P2 定义，但 P1 实际使用的是 `item_score_guess` 字段，该字段已存在于旧 SYS 格式。因此 **P1 可与 P0 并行**）
3. **P2**（决策树重构，依赖 P0 定义的常量）
4. **P3**（记忆结构，依赖 P2 的 SYS 摘要字段）

**推荐执行流**：P0 + P1 并行 → P2 → P3

### 每阶段验证

| Phase | 验证方法 |
|-------|---------|
| P0 | 发起对话 → 确认无时间提醒出现 → 确认 prompt 含深度约束文字 |
| P1 | 对话中完成条目 → 检查 `session.item_scores_initial` 写入 item_score_guess |
| P1 | 用含 insufficient 条目的对话记录调用 /api/score → 确认该条目 score=0，reason 含 "[系统钳位]" |
| P1 | 确认评分 JSON 含 symptom_evidence / severity_judgment / matched_anchor_index |
| P1 | 全局校准验证：构造 Q1 初评偏高但后续条目暗示正常的场景 → 确认 `calibration_note` 说明修正原因 |
| P2 | 连续两次模糊回答 → 确认 SYS 输出 info_sufficiency: 0/1 + missing_dimensions |
| P2 | 确认第 2 轮追问 prompt 含追问方向 |
| P2 | 确认 forced-choice 在适当轮次触发 |
| P2 | 用旧格式 SYS 测试向后兼容（`info_sufficient: true` 应映射为 `info_sufficiency: 2`） |
| P3 | 完成一条目 → 检查 session.item_summaries 是否写入 |
| P3 | 触发下一条目 → 检查 prompt 含历史摘要 |
| P3 | 确认 LLM 不会在对话中说"你之前提到过……" |
