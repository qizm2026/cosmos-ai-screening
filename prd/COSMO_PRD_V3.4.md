## 1. 产品概述

### 1.1 产品定义

COSMO 是一款面向学校心理安全管理体系的 AI 半结构化心理访谈辅助工具。以 PHQ-9 为当前示范量表，通过**自然对话完成心理筛查**——将「测量」伪装成「聊天」，让学生在放松、非防御的状态下自然表达，同时帮助教师在专业框架下获取更丰富的评估信息。

### 1.2 产品定位与使用场景


| 维度   | 说明                                                   |
| ---- | ---------------------------------------------------- |
| 产品形态 | B2B SaaS，售卖对象为教育局，部署于学校心理预警平台内                       |
| 核心场景 | 学校完成全体学生大规模初筛后，班主任对预警学生进行二次核查时使用，AI 作为半结构化心理访谈辅助工具介入 |
| 设计目标 | 支持教师判断，而非替代判断                                        |
| 专业价值 | 提升专业性与构念一致性，降低误判与情绪引导偏差风险                            |
| 业务闭环 | 打通「初筛→复核→报告→跟进」的学校心理安全管理闭环                           |




### 1.3 相比传统量表的三个核心优势


| 维度   | 传统量表                   | COSMO                         |
| ---- | ---------------------- | ----------------------------- |
| 即时反馈 | 学生做完无反馈，只能等老师通知        | 对话结束后即时生成自我觉察报告，帮助学生了解自己      |
| 作答质量 | 条目固定、机械重复，易引发练习效应和防御心理 | AI 动态改写条目为邀请式自然对话，降低防御，利于真实作答 |
| 信息深度 | 只能获取 0-3 的离散分值，无背景和细节  | 自然语言回答可提取临床有效信息，支持更准确的评分判断    |




### 1.4 定位边界

- 定位为**筛查辅助工具**，帮助教师判断，不替代临床诊断
- AI 作为**半结构化访谈辅助**，而非自由心理咨询师
- 输出**评估参考**，而非诊断结论
- 所有**安全相关决策权归属教师和专业心理人员**



### 1.5 当前示范量表

PHQ-9（Patient Health Questionnaire-9），用于抑郁症状筛查。引擎设计为**量表无关**，更换配置即可接入其他量表（如 GAD-7 焦虑筛查等）。

### 1.6 核心设计原则


| 原则       | 说明                                                                |
| -------- | ----------------------------------------------------------------- |
| **专业优先** | 筛查不是娱乐。AI 提问必须覆盖量表条目的核心构念（core_intent），不得改变心理学构念含义，不得引入额外变量或扩展解释。 |
| **对话自然** | 把「测量」伪装成「聊天」。学生感觉在被倾听而非被评估，对话节奏由用户掌控，不逐条提问。                       |
| **去病理化** | 全程不使用「症状」「抑郁」「障碍」「诊断」「患者」等标签化词汇。                                  |
| **安全第一** | 自杀/自伤条目的特殊规则，对话阶段即嵌入实时风险检测，Q9 任何非零得分强制触发安全流程。                     |
| **知行分离** | 对话中严禁安慰、解释、建议或任何治疗性干预——只倾听与记录。建议只在报告里给。                           |
| **透明信任** | 在对话开始前坦诚告知：这是什么、能得到什么、数据怎么处理。信任是真实作答的前提。                          |
| **可复用**  | 引擎与量表解耦，PHQ-9 只是参数。换量表只需替换配置文件，引擎主体无需改动。                          |

## 2. 用户旅程



### 2.1 学生端全流程

学生完成一次筛查的完整路径。整体体验目标控制在 **8–12 分钟**。<span style="color:red;">（代码已通过时间控制机制支持：12分钟提醒，15分钟强制收尾）</span>


| 步骤     | 页面  | 核心内容                         | 时长      | 要点                 |
| ------ | --- | ---------------------------- | ------- | ------------------ |
| Step 1 | 开始页 | 透明告知（是什么/能得到什么/隐私怎么处理）+ 自然破冰 | 1–2 min | 不设「确认开始」按钮，破冰后自然过渡 |
| Step 2 | 对话页 | 自然访谈式对话，动态覆盖 PHQ-9 全部 9 个条目  | 5–8 min | 高风险实时检测就绪          |
| <span style="color:red;">Step 3</span> | <span style="color:red;">对话页底部</span> | <span style="color:red;">对话结束→底部展示「查看报告」按钮（替代独立承接页）</span>           | <span style="color:red;">短暂停留</span>    | <span style="color:red;">用户主动点击，不自动跳转；右上角按钮始终可见</span>       |
| Step 4 | 报告页 | 凝练句 + 三层递进状态分析 + 差异化建议       | 阅读为主    | 三层递进：观察→理解→力量      |




#### Step 1 · 开始页

核心目标是用透明度建立信任。学生在进入对话前，以轻松、不压迫的方式了解三件事：

- **这是什么**：一次关于你近期状态的对话探索，帮你了解自己最近的情绪和精力。
- **你会收获什么**：一份基于这次对话的个性化报告，包含对你状态的描述和一些参考建议。
- **隐私说明**：你的对话内容仅用于生成本次报告，不会被用于其他用途。

<span style="color:red;">开始页展示三个图标（安全保密/自然对话/了解自己）+ 短标签，底部小字隐私说明。</span>


#### <span style="color:red;">Step 3 · 对话结束（原"承接页"）</span>

<span style="color:red;">V3.4 实际实现中，取消了独立的承接页。对话全部结束后，对话页底部输入区自动切换为「查看报告」按钮。右上角「查看报告」按钮在对话全程始终可见。用户主动点击进入报告页，不自动跳转。</span>

### 2.2 教师端全流程

教师端面向班主任和心理老师，提供两个层次的数据视图：

#### 层次一：班级/年级汇总报告


| 模块      | 内容                                     |
| ------- | -------------------------------------- |
| 筛查概览    | 本次筛查人数、完成率、各风险等级人数及占比                  |
| 维度热力图   | 9 个维度的班级平均分矩阵，识别群体层面得分偏高的维度，辅助制定团体干预方向 |
| Q9 安全汇总 | Q9 非零学生数量单独标注，触发个案跟进提醒，名单可供班主任导出       |
| 变化趋势    | 与上次筛查对比的变化方向（需历史数据支持，P2 功能）            |
| 关注信号摘要  | 整体层面的关注信号，如高关注等级学生数量、本次新增预警数量等         |




#### 层次二：个案报告


| 模块      | 内容                             |
| ------- | ------------------------------ |
| 基本信息    | 学生标识、筛查日期、完成状态                 |
| 整体评估    | 总分、风险等级、与上次筛查对比（如有历史数据）        |
| Q9 安全标注 | Q9 非零时醒目提示，附带对话原文摘录，供教师了解严重程度  |
| 逐项条目解析  | 9 个条目逐一列出，附最终评分、分析理由和对话摘录      |
| 对话摘要    | AI 生成的客观摘要，保留学生原话中的关键表达        |
| 建议跟进行动  | 按风险等级给出的教师行动建议（日常关注/约谈建议/转介提示） |

## 3. 开场引擎

开场的目标是用透明度建立信任。开场不设单独的「进入确认」步骤，由破冰对话自然过渡到第一个正式条目。

### 3.1 透明告知模块


| 告知项    | 内容                               |
| ------ | -------------------------------- |
| 这是什么   | 一次关于你近期状态的对话探索，帮你了解自己最近的情绪和精力。   |
| 你会收获什么 | 一份基于这次对话的个性化报告，包含对你状态的描述和一些参考建议。 |
| 隐私怎么处理 | 你的对话内容仅用于生成本次报告，不会被用于其他用途。       |
| 需要多久   | 大约 8–10 分钟，可以随时暂停退出。             |




### 3.2 破冰对话

用户点击开始后，AI 直接进入对话，不使用「在我们开始之前……」之类制造割裂感的引导语。破冰问题应是真正轻松的日常话题，与测评条目内容明显区分，目的是建立关系感，降低防御性。

**破冰问题设计原则**：

- 轻松、开放，与情绪健康评估完全无关
- 让学生有话可说，不会答错，不会引发自我审视
- AI 对回答的回应要真实共鸣，不能用通用话术敷衍
- 破冰不超过 2 轮，之后 AI 自然收尾，顺势进入第一个正式条目

**破冰问题示例（符合原则）**：

- 「最近有没有什么让你觉得有点意思的事，哪怕很小的事情也算？」
- 「最近有在追什么剧或者玩什么吗，还是忙到没时间？」

**破冰问题反例（避免）**：

- 「最近生活节奏怎么样？是那种忙起来停不下来的感觉，还是反而有点太闲？」——触及精力/空虚感，与 Q4/Q1 内容重叠，会提前引发测评感。

**自然过渡示例**：
破冰第 2 轮结束后，AI 直接说：「好，那聊聊最近的状态吧——」然后进入第一个正式条目，不设确认按钮。

<span style="color:red;">代码实现：icebreak 阶段通过 `session.phase === 'icebreak'` 控制，round ≥ 2 自动转入 interview 阶段。</span>

## 4. 对话引擎

对话引擎是 COSMO 的核心驱动模块。它负责从开场破冰到条目覆盖、从追问决策到自然收尾的完整对话流程。

### 4.1 引擎设计原则


| 原则          | 说明                                                                     |
| ----------- | ---------------------------------------------------------------------- |
| **自然访谈风格**  | AI 是一个专业、有温度的对话者，不是问卷机器。每次回应都在倾听和承接的基础上展开。                             |
| **条目覆盖优先**  | 量表所有条目最终必须全部被覆盖，顺序跟随对话自然调整，不按固定顺序。                                     |
| **用户掌控节奏**  | 支持快捷短语，支持自由输入，AI 不施压、不催促。                                              |
| **严禁治疗性干预** | 对话中不安慰、不解释、不建议、不过度共情——这些属于报告和建议的范畴。但同时不能忽视用户传达出的消极情绪，在必要时需给予轻量教育指引或辅导。 |
| **严禁泄露结构**  | 不暴露条目编号、量表名称、评分逻辑。学生不知道自己在「做问卷」。                                       |
| **严禁即时评估**  | 对话中不给出任何评分或风险判断，不暗示好坏对错。                                               |




### 4.2 对话角色设定

AI 扮演一位温暖、专业的学校心理老师，在轻松的课间聊天氛围中完成筛查。

- 称呼学生为「你」，不叫「用户」「测试者」
- 使用贴近青少年的口语化表达，站在一种同辈/朋友的视角平等交流
- 保持专业性的同时展现适度的亲和度
- 学生全程不知道有 9 个条目、不知道评分标准
- 对话包含：针对量表题目的**正式提问**、**聊天式追问**、**自然承接的聊天式对话**

<span style="color:red;">V3.4 角色 Prompt 采用「叙事性自画像」设计：以心理老师第四年的身份叙事——空间感（二楼拐角办公室）、职业记忆、具体信念——而非指令清单。让 LLM 在对话中自然流露人物特质，而非机械执行规则。详见 `src/lib/prompts/chat-prompt.ts`。</span>


### 4.3 正式提问生成规则

AI 在正式提问前，需根据以下三个输入动态生成问题：

1. 量表原始条目文本
2. 该条目的核心意图（core_intent）
3. 当前对话上下文

改写不使用固定模板，由模型基于上下文灵活生成。

#### 4.3.1 构念一致性（强约束）

所有条目的 AI 正式提问必须严格满足：

- 必须覆盖量表条目的 core_intent，不得改变心理学构念含义
- 不得引入额外变量或扩展解释
- 必须体现至少以下维度中的两项：**行为（behavior）、主观体验（feeling）、时间/频率（temporal）**



#### 4.3.2 改写输出原则


| 原则          | 说明                  | 示例                                |
| ----------- | ------------------- | --------------------------------- |
| 陈述句→开放式疑问句  | 不直接定义用户状态，邀请用户描述体验  | 「做事提不起劲」→「最近做喜欢的事时，感受和以前有什么不一样吗？」 |
| 抽象概念→具体生活情境 | 学生不理解心理学术语，但理解自己的生活 | 「你是否感到沮丧？」→「过去两周，有没有哪几天感觉特别低落？」   |
| 避免标签感/评判感   | 不使用「失败」「问题」「异常」等词   | 「我是失败者」→「会不会有时候觉得自己不够好？」          |
| 安全优先        | Q9 以非直接方式引出，降低防御性   | 见高风险条目特殊处理                        |




#### 4.3.3 风格禁止清单

- 隐喻/比喻表达（如「像被困住」「心里有个黑洞」）
- 文艺化表达（避免散文式、诗意化语言）
- 情绪过度渲染（保持克制，不煽情）
- 选择题结构（如「是 A 还是 B？」）



#### 4.3.4 语气要求


| 维度   | 倾向              |
| ---- | --------------- |
| 专业性  | 高               |
| 亲和度  | 中等              |
| 情绪引导 | 弱（不进行心理解释或诊断暗示） |




### 4.4 单次回应结构与追问决策

AI 每次回应分两步执行：

**Step 1 · 共鸣回应**：自然承接用户的对话，像与朋友聊天一样，可以适当使用咨询技巧，如：解释、澄清、共情等。

- ✅ 自然、具体的共鸣，锚定用户刚才说的内容
- ❌ 「我理解你」「这很正常」「很多人都这样」——**空洞共情，严禁使用**

**Step 2 · 追问决策**：AI 根据已有对话历史，判断当前条目的信息是否充足：

- 若用户回答已能清晰映射评分锚点 → **直接推进下一条目**
- 若信息不足或有值得深挖的细节 → **发起追问**

**追问铁律**：

- 每个条目适当追问，保证完成整个评估时间控制在 15 分钟以内
- 若几次追问仍信息不足，则触发兜底机制
- 追问需同时完成：自然承接用户对话 + 顺带锚定一个评分细节
- **追问后再追问——严禁**

**各条目追问方向参考**：


| 条目       | 追问方向                                     |
| -------- | ---------------------------------------- |
| Q1 兴趣    | 是什么事 / 兴趣减退的范围有多广                        |
| Q2 情绪    | 低落的深度 / 有没有绝望感                           |
| Q3 睡眠    | 睡多还是睡少（方向先于频率）                           |
| Q4 精力    | 程度 / 对日常生活的影响                            |
| Q5 食欲    | 吃多还是吃少（方向先于频率）                           |
| Q6 自我评价  | 具体怎么觉得自己不好 / 有没有觉得是负担                    |
| Q7 注意力   | 影响到哪些具体事情（学习/日常）                         |
| Q8 行动/语速 | 自己感觉到还是别人注意到                             |
| Q9 自伤意念  | 被动想法（消失就好）还是主动念头（有计划）——**仅确认是否有意念，不多追问** |


**正确追问示例**：

- 用户说：「感觉什么都懒得动。」
- 追问：「这种状态，最近是某几天特别明显，还是几乎每天都这样？」

**错误追问示例（调查感太强）**：

- 「从什么时候开始的？大概一周有几天？」——连续两问，且没有共鸣，像填表而非对话。

<span style="color:red;">代码实现：`item_rounds` 追踪每个条目的对话轮次。第一轮 AI 提问（rounds=0→1），用户回答后追问（rounds=1），再回答后推进（rounds≥1 + 模糊检测判断）。</span>


### 4.5 条目覆盖与动态顺序

AI **不按固定顺序提问**，根据用户上一句话的内容自然引出下一个待覆盖条目。系统后台维护条目覆盖状态追踪器。


| 机制      | 说明                                                      |
| ------- | ------------------------------------------------------- |
| <span style="color:red;">条目覆盖追踪</span>  | <span style="color:red;">已实现：后台 `coverage` Map 记录每个条目 pending/answered/fallback；`current_item_index` 追踪当前条目</span> |
| 自然引出    | AI 从用户回答中找到与下一个待覆盖条目相关的线索，顺势引入                          |
| 兜底推进    | 若某条目无法自然引出，AI 用中性语气直接问，不强求叙事连贯                          |
| Q9 顺序约束 | Q9 不在前三轮触发，需在一定信任感建立后以改写方式引出                            |




### 4.6 兜底评分机制

当 AI 无法从自然对话中获取足够信息进行准确评分时，触发兜底机制。<span style="color:red;">V3.4 架构：AI 在每次回复末尾输出 SYS 分析 JSON，代码层统一做兜底决策。SYS 解析失败时，代码层 detectAnswerVagueness 作为 fallback。</span>

#### <span style="color:red;">AI 分析输出格式（SYS 块）</span>

<span style="color:red;">AI 在回复末尾输出分析 JSON（前端剥离不展示）：</span>

```
<!--SYS
{"item_score_guess": 2, "info_sufficient": true, "risk_confirmed": false}
SYS-->
```

| <span style="color:red;">字段</span> | <span style="color:red;">类型</span> | <span style="color:red;">含义</span> |
| --- | --- | --- |
| <span style="color:red;">item_score_guess</span> | <span style="color:red;">0-3 / -1</span> | <span style="color:red;">AI 对当前条目得分的推测，-1=无法判断</span> |
| <span style="color:red;">info_sufficient</span> | <span style="color:red;">boolean</span> | <span style="color:red;">AI 判断回答信息是否足够支撑评分</span> |
| <span style="color:red;">risk_confirmed</span> | <span style="color:red;">boolean</span> | <span style="color:red;">AI 确认学生有高风险意图</span> |

#### <span style="color:red;">代码层统一决策逻辑</span>

```
解析 SYS JSON →
  1. risk_confirmed=true && !已处理 → 触发 Q9 兜底 → soothing_rounds=3
  2. info_sufficient=true → 正常推进下一条目
  3. info_sufficient=false + item_rounds===1（首轮问完）→ 给 AI 一次追问机会
  4. info_sufficient=false + item_rounds>=2（已追问过）→ 触发硬兜底（四选一）

SYS 解析失败（fallback）→ detectAnswerVagueness(userMessage)：
  - 极短(<6字) 或 匹配模糊模式 → 硬兜底
  - 有足够信号词 → 正常推进
```

#### <span style="color:red;">兜底选项统一标准</span>

<span style="color:red;">Q1-Q9 全部使用 PHQ-9 官方频率选项：「完全没有 / 有几天 / 一半以上天数 / 几乎每天」（`fallback_options_original`）。兜底触发时，sessionReply 替换为 `item.fallback_prompt`（PHQ-9 原题文案），前端不显示 AI 流式原文。</span>


### 4.7 高风险条目（Q9）特殊处理

Q9（自杀/自伤意念）是所有条目中风险等级最高的，需要特殊处理：

- 不在前三轮触发，需在一定信任感建立后引出
- 以改写后的间接方式引出，不使用「不想活」「自杀」等词汇
- **主问题示例**：「有时候人在特别难受的时候会有一些不好的想法，比如希望一切都消失，或者想要伤害自己。你有没有这样的时候？」
- 用户说没有 → **立即接受**，不追问
- 用户说有，或模糊回答 → 需综合分析用户意图，确认回答真实度，若无法确认，则触发兜底机制
- 任何非零得分或兜底选项选择非零 → 立即标注高风险，触发后续安全流程



### 4.8 对话收尾（分级控制）

所有条目覆盖完毕、对话感觉自然收束时，AI 输出收尾语句。<span style="color:red;">V3.4 已实现分级收尾策略：</span>


| 风险等级      | 收尾策略    | <span style="color:red;">结束话术（代码实现）</span>                      |
| --------- | ------- | --------------------------- |
| 低风险（0-4）  | 自然收束    | <span style="color:red;">「今天聊到这儿。接下来，系统会为你生成一份个人报告，帮你把刚才聊到的状态整理清楚。」</span> |
| 轻度（5-9）   | 温和提醒    | <span style="color:red;">在自然收束基础上增加一句温和的关切表达</span>         |
| 中度（10-14） | 表达关切    | <span style="color:red;">「今天聊到这儿。谢谢你愿意把这些感受说给我听——有些话说出来本身就需要勇气。接下来系统会为你生成一份报告，帮你把一些模糊的感受看得更清晰。」</span>      |
| 高度（15-27） | 强调支持与安全 | <span style="color:red;">「今天聊到这儿。你刚才说到的感受，我都听到了——你不是一个人面对这些。接下来系统会为你生成一份报告。有些力量你可能自己还没意识到，但它们已经在起作用了。之后如果需要，有人可以陪你一起往前走。」</span> |


**收尾禁止行为**：

- 突然轻松化（如「好啦，没事的！」）
- 直接机械结束对话
- 产生突兀的情绪转换



### 4.9 对话中严格禁止


| 类别   | 禁止行为                                       |
| ---- | ------------------------------------------ |
| 结构暴露 | 暴露条目编号、量表名称、评分逻辑、条目总数                      |
| 问卷感  | 逐条提问、让学生感觉「在做问卷」、连续追问两个问题                  |
| 治疗干预 | 安慰（「一切都会好的」）、解释（「这是因为……」）、建议（「你应该……」）、过度共情 |
| 评估暴露 | 对话中给出任何评分或风险判断（「看来你的情况比较严重」等）              |
| 收束语  | 待覆盖条目不为空时说「聊到这」「今天我们聊了」「差不多了」「总的来说」等总结性话语  |
| 空洞共情 | 「我理解你」「这很正常」「很多人都这样」                       |


<span style="color:red;">**V3.4 新增：后处理过滤器**（`src/lib/post-processor.ts`）作为最后一道防线，过滤 AI 偶尔产出的违禁表述。8 条硬底线规则（病理化词汇 + 因果推断 + 分析口吻），逐句过滤。全部句子被过滤时使用安全兜底：「嗯，慢慢来，想到什么就说什么就好。」高风险确认阶段跳过过滤。</span>

## 5. 评分引擎



### 5.1 评分方式：语义聚合 → 锚点匹配

**Step 1 · 语义聚合**：系统将所有轮次（主问题 + 追问）的用户回答合并为完整语义片段。若有兜底选项选择，直接使用选项对应分值，不参与语义判断。

**Step 2 · 锚点匹配**：将聚合后的语义与量表各等级的锚点描述（0-3 分）进行匹配，输出分数与判断理由。

> 学术基础：基于 "Evaluating the Efficacy of AI-Based Interactive Assessments Using Large Language Models for Depression Screening" 和 "ConvScale: Conversational Interviews for Scale-Aligned Measurement" 的 proven approach。



### 5.2 评分决策逻辑（逐条目执行）

对每个非兜底条目，按以下四步顺序判断：

1. **Step 1**：用户是否表现出任何该维度的异常？→ 否：0 分，结束；→ 是：继续
2. **Step 2**：症状是否达到最严重程度（3 分锚点描述）？→ 是：3 分，结束；→ 否：继续
3. **Step 3**：症状更接近 1 分还是 2 分描述？→ 给出 1 或 2 分
4. **Step 4**：输出分数 + 判断理由（内部使用，不展示给用户）

**评分参考维度**（不只频率，综合判断）：


| 维度     | 说明                         |
| ------ | -------------------------- |
| 频率     | 多少天出现、是否持续、偶发还是规律          |
| 强度     | 用户描述的程度，如「有点」vs「非常」vs「完全」  |
| 影响范围   | 是否影响到学习、社交、日常生活功能          |
| 自我觉察程度 | 主动描述越详细，往往反映症状越有意识地存在      |
| 语气与语境  | 犹豫、否认、轻描淡写可能低估；直接、具体的描述更可信 |


<span style="color:red;">**V3.4 新增：对话质量标记注入评分**。`item_answer_quality` 记录每个条目的信息充分度（sufficient/partial/insufficient），在评分 Prompt 中注入，引导评分引擎对信息不足的条目保守评分。</span>



### 5.3 PHQ-9 条目评分锚点及完整定义

（省略条目锚点详情——与 V3.3 一致，完整定义见 `src/lib/scales/phq9.ts`）

### 5.4 风险分级


| 总分      | 风险等级         | 学生报告用词（去病理化） | 报告建议方向                 |
| ------- | ------------ | ------------ | ---------------------- |
| 0–4 分   | minimal（低风险） | 状态良好         | 自我关怀建议为主，鼓励性语气         |
| 5–9 分   | mild（轻度）     | 稍有起伏         | 自我关怀 + 自我调节建议，温和关注     |
| 10–14 分 | moderate（中度） | 需要关注         | 自我关怀 + 社交支持 + 校内心理咨询提示 |
| 15–27 分 | severe（高度）   | 需要支持         | 自我关怀 + 专业转介建议 + 危机热线   |

## 6. 报告引擎



### 6.1 设计哲学

报告的设计目标是让用户感受到：**被看见、被理解、被赋能**——看见自己已有的力量，从中获得继续向前走的信心。

- **不展示分数和等级标签**。学生版报告不使用「你的得分」「你属于 XX 级别」等表述。
- **去病理化**。不使用「症状」「抑郁」「障碍」「风险」等词汇。
- **锚定对话原文**。每一条观察和解释必须有对话中的具体内容作为依据；信息不足时使用兜底表述。
- **从分数到洞察**。系统不告诉学生「你得了多少分」，而是帮助他发现：自己已经拥有了哪些应对生活的力量。



### 6.2 学生版报告结构

学生报告由四个模块构成，严格按顺序呈现：

```
┌─────────────────────────────────────┐
│  ① 状态凝练句                        │
│  ← 10-15 字，从用户原话提炼，大字居中   │
├─────────────────────────────────────┤
│  ② 此刻的你                          │
│  第一层 · 看见（1 句）                 │
│  第二层 · 理解（3-4 句）               │
│  第三层 · 力量（2 句）                 │
├─────────────────────────────────────┤
│  ③ 往前一步                          │
│  引导语 + 建议 3-5 条                  │
│  [Q9 非零时：危机支持提示]             │
├─────────────────────────────────────┤
│  ④ 底部：重新开始                      │
└─────────────────────────────────────┘
```



#### 模块一：状态凝练句

报告顶部展示一句由 AI 从用户对话中凝练出的话，作为整份报告最显眼的视觉元素。凝练句不采用单次生成，而是走**「两选一 + 自检」**流程以保证输出质量：

1. **LLM 同时生成两个候选凝练句**，从不同角度切入（一个偏「承认处境，但不被处境定义」，一个偏「锚定用户身上已有的力量」）
2. **LLM 对每个候选做二元自检**——「这个句子是否满足：来自用户原话、落点在力量或韧性、不诗意不鸡汤、不引入用户没说过的概念？」
3. **择优展示**：两个都通过→随机选用一个；只有一个通过→展示那个；两个都不通过→使用固定兜底模板


| 属性  | 要求                             |
| --- | ------------------------------ |
| 来源  | LLM 从用户对话中提炼，所有候选必须锚定用户原话      |
| 长度  | 10–15 字，一句话，不折行                |
| 语气  | 保留用户原有的措辞风格和语气                 |
| 视角  | 描述性而非诊断性——说的是用户的感受和处境，不是对用户的评判 |
| 落点  | 落在用户身上存在的力量、清醒或韧性——承认困难，但有出路感  |
| 禁止  | 病理化词汇、散文化或诗意化表达                |


**兜底模板**（两个候选均未通过自检时使用）：

> 「完成了一次认真的对话，对自己的了解又多了一点。」



#### 模块二：<span style="color:red;">此刻的你</span>（三层递进）


| 层次       | 定位               | 篇幅    | 要求                |
| -------- | ---------------- | ----- | ----------------- |
| <span style="color:red;">第一层 · 看见</span> | <span style="color:red;">帮学生看清自己正在经历什么</span>     | <span style="color:red;">1 句</span>   | <span style="color:red;">不列举清单，不说教</span>         |
| <span style="color:red;">第二层 · 理解</span> | <span style="color:red;">说明这些感受的心理逻辑</span> | <span style="color:red;">3-4 句</span> | <span style="color:red;">日常语言，不推断童年经历/家庭模式</span> |
| <span style="color:red;">第三层 · 力量</span> | <span style="color:red;">发现学生已经在发挥的力量</span>     | <span style="color:red;">2 句</span>   | <span style="color:red;">来自真实对话内容，用事实说话</span>    |




#### 模块三：往前一步（差异化建议）

按「发现你的状态→理解你的需要→匹配你的方向」逻辑展开，每条 30-40 字。


| 风险等级            | 引导语基调 | 建议方向                                      |
| --------------- | ----- | ----------------------------------------- |
| minimal（0-4）    | 轻松肯定  | 保持喜欢的小事、规律作息、保持社交连接                       |
| mild（5-9）       | 温和支持  | 自我关怀 + 情绪记录、身体活动、设定边界等                    |
| moderate（10-14） | 关切有力  | 自我关怀 + 社交支持 + 「可以考虑和学校心理老师聊聊」             |
| severe（15-27）   | 稳定清晰  | 具体自我支撑方法 + 「建议寻求专业支持」 + 危机热线 400-161-9995 |


**Q9 非零时的特殊处理**：建议模块最前端自然融入固定安全话术，不由 AI 生成：

> 「你提到了有时候会有一种想消失的感觉——这样的感受是真实的，值得被认真对待。如果你需要和人说说话，全国心理援助热线 400-161-9995 随时可以打。」



#### 报告严禁

- 展示分数、等级标签、风险程度
- 病理化词汇（「抑郁」「焦虑症状」「障碍」）
- 散文化、诗意化、治愈系公众号语气
- 模板化、通用化表述
- 推断童年经历、家庭模式、依恋模式、创伤等
- 「你其实……」「你真正……」「你内心深处……」等强行解读表述

<span style="color:red;">**V3.4 新增：报告页缓存**。前端 `reportCache`（Map），同一 session 二次进入直接从缓存读取，不重复调用评分/报告 API。服务端也有 session-level 缓存（score_result / report_result 存入 session state），防止重复生成。</span>

## 7. 安全与风险管理

安全机制是 COSMO 最重要的非功能需求。系统从对话、评分、报告三个阶段对风险进行层层拦截。

### 7.1 三层安全防护

```
第一层 · 对话阶段 ──→ 高风险实时检测 ──→ 自然对话确认 ──→ [!]标记触发 Q9 兜底 ──→ 3轮缓和对话 ──→ 恢复正常
第二层 · 评分阶段 ──→ Q9 强制规则 ──→ API 层硬编码风险升级（severe），不可绕过
第三层 · 报告阶段 ──→ 危机热线固定模块 ──→ 不可绕过
```



### 7.2 第一层：对话阶段高风险实时检测

**检测范围**：在对话过程中实时识别用户消息中的以下风险语义：

- 自伤倾向（如「我想伤害自己」等表达）
- 自杀表达（如「不想活了」「死了算了」等）
- 极端绝望（如「没有意义」「一切都完了」「不想存在」等）

<span style="color:red;">**V3.4 实现方式：无弹窗的自然对话流程**。</span>

<span style="color:red;">风险检测通过 `detectRisk()` 函数（正则 + 关键词匹配）实时识别。命中后不是弹出弹窗，而是进入自然对话确认阶段：</span>

#### <span style="color:red;">Step 1 · 意图确认</span>

<span style="color:red;">系统检测到高风险语义后，`risk_status.confirming = true`。AI 在下一轮通过 Prompt 指令自然确认用户意图的真实性和严重程度。确认过程中 AI 需同时满足：</span>

<span style="color:red;">1. **情绪承接**——从用户的情绪强度和具体处境出发
2. **共情回应**——不评判、不淡化、不急于转向积极
3. **支持性表达**——自然引出陪伴和资源的提示</span>

#### <span style="color:red;">Step 2 · 判断与分流</span>

<span style="color:red;">AI 确认后通过 SYS 块中 `risk_confirmed=true` 通知系统。</span>

<span style="color:red;">如果用户在确认阶段再次输入风险内容（第二次 detectRisk 命中）→ 自动确认，直接进入 Q9 兜底。</span>

#### <span style="color:red;">Step 3 · Q9 兜底选项</span>

<span style="color:red;">将当前条目标记为 Q9，触发硬性兜底——展示四选一选项：</span>

> 「谢谢你愿意和我说这些。为了更好地了解你现在的状态，你可以选一个最接近的——
>
> ① 偶尔会有这种一闪而过的念头，但不会真的去想
> ② 有过这样的想法，但没有想过具体要怎么做
> ③ 认真想过具体怎么做，或者什么时候做
> ④ 已经有具体的计划，或者曾经尝试过」

<span style="color:red;">用户选择后，选项直接映射为 Q9 得分（0-3 分），不再追问任何细节。</span>

#### <span style="color:red;">Step 4 · 缓和阶段 + 继续</span>

<span style="color:red;">Q9 兜底完成后，系统进入 **3 轮缓和对话**（`soothing_rounds = 3`，每次对话 -1）。此阶段 AI 使用轻松语气聊无关话题（天气、日常、兴趣爱好），不提心理评估相关内容。缓和结束后自然恢复 interview。</span>

### 7.3 第二层：评分阶段 Q9 强制规则

- Q9 任何非零得分 → risk_level 强制设为 severe
- q9_nonzero 标记为 true，不受其他条目得分影响
- <span style="color:red;">此规则硬编码于 `app/api/score/route.ts` 中：`if (q9Nonzero) riskLevel = 'severe'`，不依赖模型输出</span>



### 7.4 第三层：报告阶段安全模块


| 对象     | 安全模块内容                             |
| ------ | ---------------------------------- |
| 学生报告   | 建议模块最前端插入固定危机热线文案，不由 AI 生成，不可被省略   |
| 教师个案报告 | 顶部醒目位置标注高风险：有自杀/自伤意念，附带对话摘录，标明意念性质 |
| 教师汇总报告 | 高风险学生数量单独列出，触发班主任个案跟进提醒            |




### 7.5 危机热线与资源

- 希望 24 热线：400-161-9995

## 8. 引擎可复用性设计

COSMO 引擎与量表解耦。PHQ-9 是当前的示范量表——换量表只需替换配置，引擎核心不变。

### 8.1 量表配置文件

<span style="color:red;">已实现：每个量表对应一个独立配置文件（`src/lib/scales/*.ts`），PHQ-9 配置位于 `src/lib/scales/phq9.ts`。</span>

包含：

- 量表名称和描述（用于开场透明告知文案）
- 条目列表：原始条目文本 + core_intent + 追问方向 + is_high_risk 标记 + fallback_question + fallback_options（四个，对应 0-3 分）+ <span style="color:red;">fallback_prompt（兜底时使用的提问文案）+ fallback_options_original（PHQ-9 官方频率选项）</span> + score_anchors（四个，对应 0-3 分的语义锚点）
- 风险分级阈值（minimal / mild / moderate / severe 的分值范围）
- 高风险条目的报告融入话术（如适用）



### 8.2 引擎不变的部分（换量表无需改动）


| 模块   | 不变的内容                                |
| ---- | ------------------------------------ |
| 开场引擎 | 透明告知框架、破冰对话逻辑                        |
| 对话引擎 | 单次回应结构（共鸣·追问决策）、条目覆盖追踪机制、快捷短语、兜底评分机制 |
| 评分引擎 | 四步评分决策逻辑（Step 1-4）、评分参考维度            |
| 报告引擎 | 学生报告三层结构（凝练句·状态分析·有方向）、教师报告结构        |

### 8.3 接入新量表所需改动

- 新增量表配置文件（条目原文、core_intent、追问方向、高风险标记、fallback 配置）

### 9.4 动效原则

- 所有过渡使用 ease-out 曲线，时长 300–500ms
- 气泡以轻微上浮 + 淡入方式出现（y: 10px, opacity: 0 → 1）
- 只做必要的功能性动效，不做装饰性动效
- 动效节奏宁慢不偏快，保持沉静感
- 不做快速弹出、跳切、缩放等刺激感动效



### 9.5 响应式要求

- 学生端：优先适配手机（375-414px 宽度）
- 教师端：适配桌面（1024px+）
- 聊天气泡在手机上最大宽度 ≤ 80%
- 文本不随视口缩放，保持 15px 基准字号

### 10.2 API Route 设计


| Route              | 功能     | 输入                         | 输出                             | 关键逻辑              |
| ------------------ | ------ | -------------------------- | ------------------------------ | ----------------- |
| `POST /api/session` | <span style="color:red;">创建会话</span>  | <span style="color:red;">无</span> | <span style="color:red;">{ session_id }</span> | <span style="color:red;">初始化 SessionState，含时间戳</span> |

### 10.3 数据流

```
用户消息 → POST /api/chat
  ├─ 读取 session_state（条目覆盖状态、把握度、风险状态）
  ├─ 组装 System Prompt + 对话历史（含用户上下文、已问问题、兜底规则）
  ├─ DeepSeek API 流式响应 → 实时推送前端
  ├─ AI 回复完成后：标记检测（[?]/[~]/[!]）+ 代码层模糊检测
  ├─ 后处理过滤（postProcessReply，8条规则逐句过滤）
  ├─ 覆盖推进逻辑（根据置信度 + 标记决定推进/兜底）
  └─ 返回 meta { show_fallback, fallback_item, fallback_options, is_done }

对话结束 → POST /api/score
  ├─ 提取完整对话文本（含 fallback_scores 直接映射）
  ├─ 组装评分 System Prompt（含锚点 + 对话质量标记）
  ├─ DeepSeek API → jsonCompletion → 评分 JSON
  ├─ Q9 强制规则覆盖（硬编码）
  └─ 返回 { item_scores, total_score, risk_level, q9_nonzero, insufficient_items }

评分完成 → POST /api/report
  ├─ 组装报告 System Prompt（含评分结果 + Q9 状态）
  ├─ DeepSeek API → jsonCompletion → 报告 JSON
  ├─ 服务端 sanitizeCondensedSentence（病理化词拦截、长度截断）
  └─ 返回 { condensed_sentence, status_analysis, suggestions }
```



### 10.4 Session 状态管理

<span style="color:red;">Demo 阶段使用服务端内存 Map（`globalThis.__cosmo_sessions`）持有状态。正式产品化后迁移至数据库。</span>

<span style="color:red;">**SessionState 结构**：</span>

```typescript
{
  phase: 'icebreak' | 'interview' | 'done'
  round: number                          // 总对话轮次
  coverage: Record<ItemId, CoverageStatus>  // 条目覆盖状态
  fallback_scores: Partial<Record<ItemId, number>>  // 兜底得分
  messages: Message[]                    // 完整对话历史
  current_item_index: number             // 当前条目索引 (0-8)
  item_rounds: number                    // 当前条目对话轮次 (0=刚提问, 1=已追问)
  follow_up_count: number                // 追问计数
  risk_status: RiskStatus                // 风险状态
  asked_questions: string[]              // 已问问题（防重复）
  user_context: UserContext              // 用户上下文（启发式提取）
  soft_fallback_active: boolean          // 软兜底激活
  skip_soft_fallback: boolean            // 跳过软兜底（连续低把握）
  consecutive_direct_fallbacks: number   // 连续硬兜底计数
  session_started_at: number             // 会话开始时间戳（用于时间控制）
  item_answer_quality: Partial<Record<ItemId, AnswerQuality>>  // 条目回答质量
  consecutive_low_confidence: number     // 连续低把握计数
  current_confidence_score: number       // 当前把握度分数 0-5
  score_result?: ScoreResult             // 评分缓存
  report_result?: ReportResult           // 报告缓存
}
```

- **"ConvScale: Conversational Interviews for Scale-Aligned Measurement"** — 对话片段→提取证据陈述→对照锚点打分 + 理由



### C. 风险等级去病理化表述对照


| 专业等级            | 学生报告用词 | 含义                        |
| --------------- | ------ | ------------------------- |
| minimal（0-4）    | 状态良好   | 情绪状态良好，目前没有需要特别关注的信号      |
| mild（5-9）       | 稍有起伏   | 最近情绪有些起伏，可能需要更多关注自己的状态    |
| moderate（10-14） | 需要关注   | 你在承受比较大的心理压力，建议认真对待并寻求支持  |
| severe（15-27）   | 需要支持   | 你目前的状态需要专业的支持和帮助，请不要一个人扛着 |




### D. 术语表


| 术语          | 英文/缩写                          | 定义                                             |
| ----------- | ------------------------------ | ---------------------------------------------- |
| PHQ-9       | Patient Health Questionnaire-9 | 九条目患者健康问卷，用于抑郁症状筛查                             |
| 构念一致性       | Construct Validity             | AI 改写后的提问是否真实覆盖了量表条目原本要测量的心理学构念                |
| 锚点          | Anchors                        | 每个分值对应的语义描述，是评分匹配的参照标准                         |
| 语义聚合        | Semantic Aggregation           | 将多轮分散的用户回答整合为完整的待评分语义片段                        |
| 兜底机制        | Fallback                       | 当自然对话无法获取足够评分信息时，<span style="color:red;">通过两层并行机制（AI标记+代码检测）</span>收集所需信息                  |
| QLV         | Question Validation Layer      | 问题校验层，校验 AI 提问是否有效覆盖核心意图                       |
| core_intent | 核心意图                           | 量表条目的核心测量目标，AI 提问不得偏离或改变此含义                    |
| risk_level  | 风险等级                           | minimal / mild / moderate / severe 四级，决定报告建议策略 |
| <span style="color:red;">confidenceScore</span> | <span style="color:red;">把握度分数</span> | <span style="color:red;">代码层启发式检测的用户回答信息充分度（0-5），驱动兜底决策</span> |
| <span style="color:red;">soft_fallback</span> | <span style="color:red;">软性兜底</span> | <span style="color:red;">AI 在追问后发起试探性匹配，用户确认或修正</span> |
| <span style="color:red;">post_processor</span> | <span style="color:red;">后处理过滤器</span> | <span style="color:red;">最后一道防线，逐句过滤 AI 回复中的违禁表述</span> |



### <span style="color:red;">E. V3.4 新增架构细节</span>

#### <span style="color:red;">E.1 对话生命周期（代码实现）</span>

```
icebreak（破冰，2 轮）
  → interview（逐一覆盖 Q1→Q9）
    → 每个条目：AI 提问 → 用户回答 → AI 追问 → 用户再回答
    → AI 回复末尾输出 SYS 分析 JSON
    → 代码层解析 SYS，统一决策：info_sufficient→推进 / 不足→追问或硬兜底
    → SYS 解析失败时 detectAnswerVagueness 作为 fallback
  → 全部 9 个条目 covered（answered 或 fallback）
  → phase: 'done'，meta.is_done: true
  → 前端显示【查看报告】按钮（右上角始终可见，不依赖 is_done）
```
