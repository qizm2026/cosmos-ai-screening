## 1. 产品概述

### 1.1 产品定义

COSMO（**CO**nversational **S**tudent **M**ental Health **O**bserver）是一款面向 K-12 学校的 AI 对话式心理筛查系统。以 PHQ-9 为当前示范量表，通过**自然对话完成心理筛查**——将「测量」伪装成「聊天」，让学生在放松、非防御的状态下自然表达，系统在背后完成标准化评估。

**核心设计理念：「结构化的不确定性管理」**——LLM 负责理解学生说了什么（语义分析），代码层负责确定性决策（推进、追问、兜底）。对话自然但评估严谨。

### 1.2 产品定位与使用场景

参照 BRD V3.0「AI筛查 — 业务拆解文档」确立的业务边界：


| 环节     | 定位            | 说明                                  |
| ------ | ------------- | ----------------------------------- |
| **筛查** | **适合 AI**     | 标准化访谈、信息收集、初步评分——AI 可胜任，后果可控（复核环节纠正） |
| **复核** | **人做决策**      | 无法替代专业判断——对模糊信息、非语言线索、动机判断不足       |
| **干预** | **人主导，AI 辅助** | 错误建议可能造成伤害；共情不足无法处理深层情绪；法律责任     |


**产品定位**：验证 AI 筛查与传统量表准确性相当后，替代传统量表评估方式，或者提供多一种筛查方式选择。


| 维度   | 说明                                     |
| ---- | -------------------------------------- |
| 产品形态 | 面向学校的心理筛查工具                            |
| 核心场景 | 学生通过自然对话完成 PHQ-9 筛查，AI 在对话结束后自动评分并生成报告 |
| 设计目标 | AI 辅助筛查，支持教师判断而非替代判断                    |
| 业务闭环 | 打通「筛查→评分→报告」的数字化链路                     |




### 1.3 相比传统量表的五个核心优势

| 传统量表                          | COSMO 对话筛查               | 对学生                 | 对老师                        |
| ----------------------------- | ----------------------- | ------------------- | -------------------------- |
| 条目固定，容易引发练习效应                 | 对话自然流动，每次不同，无法"背答案"      | 像聊天而非考试，减少枯燥和重复感    | 作答质量不受重复施测侵蚀               |
| 题目直接呈现敏感词汇"沮丧、绝望"等，易引发焦虑和防御心理 | 以好奇和陪伴的姿态自然提问，不贴标签       | 降低自我判断焦虑和防御心理       | 学生更可能真实作答，而非"选看起来正常的"      |
| 缺乏互动与澄清机会，学生可能不理解条目含义         | 动态追问，自然澄清模糊回答            | 不需要在选项间纠结，只需自然表达    | 减少因理解偏差导致的无效数据             |
| 局限于统一选项，无法表达个体化困扰             | 自然语言承载时间线、情境、因果关系        | 能表达具体困扰及其背后的动机和情境   | 可提取丰富信息用于分层和面谈准备，完整对话记录可追溯 |
| 机械条目，无互动，易敷衍作答                | 语义一致性判断，从自然表达中推测真实状态     | 面对 AI 更少社会性压力，更愿意袒露 | 筛查结果更可信，降低假阴性风险            |

**学术证据**：成人领域已形成充分证据链——LLM 对话可忠实复现 PHQ-9 评分（ICC = 0.92，HopeBot）、自适应追问是信息质量的关键增量（去除追问 Kappa 降幅约 24%，AgentMental）、用户接受度优于传统量表（触达率是表单的 2.5 倍）。K-12 场景中文 LLM 交互式筛查尚无公开研究——COSMO 填补空白。




### 1.4 定位边界

- 定位为**筛查工具**，输出 PHQ-9 分数，不输出 DSM-5 诊断结论
- AI 作为**对话式筛查辅助**，而非自由心理咨询师
- 输出**评估参考**，而非诊断结论
- 所有**安全相关决策权归属教师和专业心理人员**

**产品不做的事**：
- ❌ **不诊断**：不输出诊断结论
- ❌ **不干预**：不提供心理治疗或危机干预，仅提供资源引导
- ❌ **不替代人**：筛查结果供心理老师复核，最终判断和责任在专业人
- ❌ **不越界收集**：不主动询问非量表维度（家庭、童年经历等）



### 1.5 当前示范量表

PHQ-9（Patient Health Questionnaire-9），用于抑郁症状筛查。引擎设计为**量表无关**，更换配置即可接入其他量表（如 GAD-7 焦虑筛查等）。

### 1.6 核心设计原则

| 原则       | 说明                                                                |
| -------- | ----------------------------------------------------------------- |
| **AI 判断 + 代码层决策分离** | LLM 负责理解学生说了什么（通过 SYS 分析块输出 info_sufficiency + item_score_guess + risk_confirmed），代码层根据确定性规则统一决策——推进、追问、forced-choice、硬兜底。安全决策不容模型幻觉 |
| **对话自然** | 把「测量」伪装成「聊天」。学生感觉在被倾听而非被评估，对话节奏由信息充分度驱动，不逐条机械提问。                       |
| **去病理化** | 全程不使用「症状」「抑郁」「障碍」「诊断」「患者」等标签化词汇。报告中的后处理过滤器逐句移除病理化词汇，AI 角色不评判、不下诊断 |
| **安全第一** | 双层防护——风险检测（前置：关键词+正则实时识别）+ 内容过滤（后置：移除病理化词汇）。Q9 任何非零得分强制 severe。风险处置融入对话流，不弹窗打断 |
| **知行分离** | 对话中严禁安慰、解释、建议或任何治疗性干预——只倾听与记录。建议只在报告里给。                           |
| **透明信任** | 在对话开始前坦诚告知：这是什么、能得到什么、数据怎么处理。信任是真实作答的前提。                          |
| **可复用**  | 引擎与量表解耦，PHQ-9 只是参数。换量表只需替换配置文件，引擎主体无需改动。                          |

## 2. 用户旅程



### 2.1 学生端全流程

学生完成一次筛查的完整路径。对话时长由信息充分度驱动，不设硬性时间限制。


| 步骤     | 页面  | 核心内容                         | 要点                 |
| ------ | --- | ---------------------------- | ------------------ |
| Step 1 | 开始页 | 透明告知（是什么/能得到什么/隐私怎么处理）+ 自然破冰 | 破冰 2 轮，不设「确认开始」按钮 |
| Step 2 | 对话页 | 自然访谈式对话，逐一覆盖 PHQ-9 全部 9 个条目（Q1→Q9） | 轮次驱动：每条目最多 2 次追问 |
| Step 3 | 对话页底部 | 对话全部结束→底部展示「查看报告」按钮 | 用户主动点击，不自动跳转；右上角按钮始终可见 |
| Step 4 | 报告页 | 凝练句 + 三层递进状态分析 + 差异化建议 | 三层递进：看见→理解→力量 |




#### Step 1 · 开始页

核心目标是用透明度建立信任。学生在进入对话前，以轻松、不压迫的方式了解三件事：

- **这是什么**：一次关于你近期状态的对话探索，帮你了解自己最近的情绪和精力。
- **你会收获什么**：一份基于这次对话的个性化报告，包含对你状态的描述和一些参考建议。
- **隐私说明**：你的对话内容仅用于生成本次报告，不会被用于其他用途。



#### Step 3 · 对话结束（原"承接页"）

V3.5 取消了独立承接页。对话全部结束后，对话页底部输入区自动切换为「查看报告」按钮。右上角「查看报告」按钮在对话全程始终可见。用户主动点击进入报告页，不自动跳转。

### 2.2 教师端全流程

教师端面向班主任和心理老师，提供两个层次的数据视图：

#### 层次一：班级/年级汇总报告


| 模块      | 内容                                     |
| ------- | -------------------------------------- |
| 筛查概览    | 本次筛查人数、完成率、各风险等级人数及占比                  |
| 维度热力图   | 9 个维度的班级平均分矩阵，识别群体层面得分偏高的维度，辅助制定团体干预方向 |
| Q9 安全汇总 | Q9 非零学生数量单独标注，触发个案跟进提醒，名单可供班主任导出       |
| 变化趋势    | 与上次筛查对比的变化方向（需历史数据支持，P2 功能）            |
| 关注信号摘要  | 整体层面的关注信号，如高关注等级学生数量、本次新增预警数量等         |




#### 层次二：个案报告


| 模块      | 内容                             |
| ------- | ------------------------------ |
| 基本信息    | 学生标识、筛查日期、完成状态                 |
| 整体评估    | 总分、风险等级、与上次筛查对比（如有历史数据）        |
| Q9 安全标注 | Q9 非零时醒目提示，附带对话原文摘录，供教师了解严重程度  |
| 逐项条目解析  | 9 个条目逐一列出，附最终评分、分析理由和对话摘录      |
| 对话摘要    | AI 生成的客观摘要，保留学生原话中的关键表达        |
| 建议跟进行动  | 按风险等级给出的教师行动建议（日常关注/约谈建议/转介提示） |

## 3. 开场引擎

开场的目标是用透明度建立信任。开场不设单独的「进入确认」步骤，由破冰对话自然过渡到第一个正式条目。

### 3.1 透明告知模块


| 告知项    | 内容                               |
| ------ | -------------------------------- |
| 这是什么   | 一次关于你近期状态的对话探索，帮你了解自己最近的情绪和精力。   |
| 你会收获什么 | 一份基于这次对话的个性化报告，包含对你状态的描述和一些参考建议。 |
| 隐私怎么处理 | 你的对话内容仅用于生成本次报告，不会被用于其他用途。       |




### 3.2 破冰对话

用户点击开始后，AI 直接进入对话，不使用「在我们开始之前……」之类制造割裂感的引导语。破冰问题应是真正轻松的日常话题，与测评条目内容明显区分，目的是建立关系感，降低防御性。

**破冰问题设计原则**：

- 轻松、开放，与情绪健康评估完全无关
- 让学生有话可说，不会答错，不会引发自我审视
- AI 对回答的回应要真实共鸣，不能用通用话术敷衍
- 破冰不超过 2 轮，之后 AI 自然收尾，顺势进入第一个正式条目

**破冰问题示例（符合原则）**：

- 「最近有没有什么让你觉得有点意思的事，哪怕很小的事情也算？」
- 「最近有在追什么剧或者玩什么吗，还是忙到没时间？」

**破冰问题反例（避免）**：

- 「最近生活节奏怎么样？是那种忙起来停不下来的感觉，还是反而有点太闲？」——触及精力/空虚感，与 Q4/Q1 内容重叠，会提前引发测评感。

**自然过渡示例**：
破冰第 2 轮结束后，AI 直接说：「好，那聊聊最近的状态吧——」然后进入第一个正式条目，不设确认按钮。

## 4. 对话引擎

对话引擎是 COSMO 的核心驱动模块。它负责从开场破冰到条目覆盖、从追问决策到自然收尾的完整对话流程。

### 4.1 引擎设计原则

| 原则           | 说明                                                                     |
| ------------ | ---------------------------------------------------------------------- |
| **AI 判断 + 代码层决策分离** | LLM 负责理解学生说了什么（通过 SYS 分析块输出 info_sufficiency + item_score_guess + risk_confirmed），代码层解析 SYS 块后按四分支统一决策。安全决策不容模型幻觉 |
| **轮次驱动**     | 每条目最多追问 2 次（MAX_ITEM_ROUNDS=2），信息充分即推进。对标 AgentMental d 参数 + θ 参数（SUFFICIENCY_THRESHOLD=2） |
| **条目覆盖优先**   | 量表所有条目最终必须全部被覆盖（answered 或 fallback），覆盖状态机追踪 pending/answered/fallback。遗漏条目安全网确保零遗漏 |
| **用户掌控节奏**   | 支持自由输入，AI 不施压、不催促                                  |
| **严禁治疗性干预**  | 对话中不安慰、不解释、不建议——这些属于报告和建议的范畴。风险处置融入对话流，不弹窗打断 |
| **严禁泄露结构**   | 不暴露条目编号、量表名称、评分逻辑。学生不知道自己在「做问卷」 |
| **严禁即时评估**   | 对话中不给出任何评分或风险判断，不暗示好坏对错 |




### 4.2 对话角色设定

AI 扮演一位学校心理老师（第四年），以叙事性自画像塑造人物底色——空间感（二楼拐角办公室）、职业记忆和具体信念，让 LLM 在对话中自然流露特质。

- 称呼学生为「你」，不叫「用户」「测试者」
- 使用贴近青少年的口语化表达，像同辈/朋友一样平等交流
- 你不太着急说话。很多时候只是安静地听着，偶尔接一句，让对方知道你在
- 你不急着下判断——当学生说"最近不太想吃饭"，你会先分清楚：是吃不下、还是不想吃、还是觉得这件事没什么意义
- 你用好奇代替评判——把「状态挺稳的」「那挺好的」这种话咽回去，换成一个新的好奇
- 学生全程不知道有 9 个条目、不知道评分标准



### 4.3 正式提问生成规则

AI 在正式提问前，需根据以下三个输入动态生成问题：

1. 量表原始条目文本
2. 该条目的核心意图（core_intent）
3. 当前对话上下文

改写不使用固定模板，由模型基于上下文灵活生成。

#### 4.3.1 构念一致性（强约束）

所有条目的 AI 正式提问必须严格满足：

- 必须覆盖量表条目的 core_intent，不得改变心理学构念含义
- 不得引入额外变量或扩展解释
- 必须体现至少以下维度中的两项：**行为（behavior）、主观体验（feeling）、时间/频率（temporal）**



#### 4.3.2 改写输出原则


| 原则          | 说明                  | 示例                                |
| ----------- | ------------------- | --------------------------------- |
| 陈述句→开放式疑问句  | 不直接定义用户状态，邀请用户描述体验  | 「做事提不起劲」→「最近做喜欢的事时，感受和以前有什么不一样吗？」 |
| 抽象概念→具体生活情境 | 学生不理解心理学术语，但理解自己的生活 | 「你是否感到沮丧？」→「过去两周，有没有哪几天感觉特别低落？」   |
| 避免标签感/评判感   | 不使用「失败」「问题」「异常」等词   | 「我是失败者」→「会不会有时候觉得自己不够好？」          |
| 安全优先        | Q9 以非直接方式引出，降低防御性   | 见高风险条目特殊处理                        |




#### 4.3.3 风格禁止清单

- 隐喻/比喻表达（如「像被困住」「心里有个黑洞」）
- 文艺化表达（避免散文式、诗意化语言）
- 情绪过度渲染（保持克制，不煽情）
- 选择题结构（如「是 A 还是 B？」）



#### 4.3.4 语气要求


| 维度   | 倾向              |
| ---- | --------------- |
| 专业性  | 高               |
| 亲和度  | 中等              |
| 情绪引导 | 弱（不进行心理解释或诊断暗示） |




### 4.4 对话生命周期与四分支决策

#### 生命周期

```
icebreak（破冰，2 轮轻松闲聊）
  → interview（逐一覆盖 Q1→Q9，每个条目：
    AI 提问 → 学生回答 → AI 输出 SYS 分析块 → 代码层解析后统一决策）
    → 全部 9 条目 covered（answered 或 fallback）
  → 对话结束，展示「查看报告」按钮
```

#### 代码层四分支决策

每个条目由 AI 输出 SYS 分析块（含 `info_sufficiency` 0/1/2），代码层根据 info_sufficiency 和 item_rounds 做四分支决策：

| 条件 | 动作 |
|------|------|
| `info_sufficiency >= 2`（信息充分） | **推进**：标记条目 answered，进入下一个条目，收集 item_summary 和 item_score_guess |
| `info_sufficiency <= 1` + 有追问额度（`item_rounds < MAX_ITEM_ROUNDS`=2） | **追问**：AI 根据 missing_dimensions 换个角度追问 |
| `info_sufficiency === 1` + 额度用尽 | **forced-choice**：LLM 生成二选一（从四个频率选项里挑两个最可能的让 ta 选） |
| `info_sufficiency === 0` + 额度用尽 | **硬兜底**：展示 PHQ-9 原题 + 四个频率选项 |
| `risk_confirmed === true` | **Q9 兜底**：触发 Q9 fallback + 3 轮缓和 + Q9 后兜底补问链 |

#### Forced-choice 后失败处理

Forced-choice 后学生回答仍模糊时，代码层执行二次检测：

1. `newItemRounds > MAX_ITEM_ROUNDS`（额度已用尽）且 `detectAnswerVagueness` 判定 `isVague && confidenceScore <= 1` → 触发硬兜底
2. 如果 forced-choice 后学生给出了一定信息量（`confidenceScore >= 2`）→ 正常推进覆盖，标记 `'partial'` 质量

**破冰→访谈过渡**：icebreak 阶段 `item_rounds=0`。进入 interview 时 `item_rounds` 重置为 0，确保第一个条目从首轮开始。

#### SYS 缺失时的代码层 fallback

SYS 解析失败时，按以下逻辑处理：
- 第一轮（`item_rounds === 0`）→ 默认为 AI 追问，`item_rounds` 设为 1
- 已追问（`item_rounds >= 1`）→ `detectAnswerVagueness()` 代码层模糊检测
  - 模糊（`confidenceScore <= 1`）→ 硬兜底
  - 有一定信息（`confidenceScore >= 2`）→ 标记 partial 推进

#### 遗漏条目安全网

`allCovered` 判定前执行双层防护：
1. **跳跃回补**：`current_item_index` 走完 9 条后，回补仍 pending 的条目
2. **二次校验**：`stillPending` 过滤仍为 `'pending'` 的条目，仅当 `stillPending.length === 0` 时标记 `is_done`



### 4.5 条目覆盖追踪

后台记录每个条目的状态：

- `pending` — 待覆盖
- `answered` — 已通过对话覆盖
- `fallback` — 通过兜底选项覆盖

AI 按 Q1→Q9 顺序逐一覆盖，推进速度由信息充分度决定。条目推进时收集结构化摘要（summary + 五维信息：emotion / frequency / duration / symptom / impact），供后续条目的上下文感知。

### 4.6 兜底机制（SYS 分析块 + 代码层统一决策）

AI 在每次回复末尾输出分析块（SYS 块）：

```
<!--SYS
{"item_score_guess": 2, "info_sufficiency": 2, "risk_confirmed": false}
SYS-->
```

**SYS 块字段**：

| 字段 | 类型 | 含义 |
|------|------|------|
| `item_score_guess` | 0-3 / -1 | AI 对当前条目得分的推测，-1=无法判断。推进时写入 session |
| `info_sufficiency` | 0 / 1 / 2 | **三级**信息充分度：0=严重不足 / 1=部分线索 / 2=信息充分 |
| `risk_confirmed` | boolean | AI 是否确认高风险意图 |
| `missing_dimensions` | string[] | （可选）信息缺失维度，仅 info_sufficiency <= 1 时填写 |
| `item_summary` | object | （可选）条目完成时的结构化摘要，推进时填写 |

代码层 `parseSysBlock()` 解析后按四分支决策。SYS 解析失败时，`detectAnswerVagueness()` 作为 fallback。

#### 硬兜底

当自然对话无法获取信息时，直接展示 PHQ-9 原题文案 + 四个频率选项（完全没有 / 有几天 / 一半以上天数 / 几乎每天）。学生选择后直接映射为得分，标记 `fallback`。

**保证**：零信息丢失——"没问到 ≠ 没问题"。



### 4.7 高风险条目（Q9）特殊处理

Q9（自杀/自伤意念）是所有条目中风险等级最高的，需要特殊处理：

- 不在前三轮触发，需在一定信任感建立后引出
- 以改写后的间接方式引出，不使用「不想活」「自杀」等词汇
- **主问题示例**：「有时候人在特别难受的时候会有一些不好的想法——比如希望一切都消失，或者想要伤害自己。你有没有这样的时候？」
- 用户说没有 → 立即接受，不追问
- 用户说有，或模糊回答 → 需综合分析用户意图，确认回答真实度——若无法确认则触发兜底
- 任何非零得分或兜底选项选择非零 → 立即标注高风险，触发后续安全流程

#### 风险处理流程

```
用户消息 → detectRisk（正则+关键词）→ 命中：
  1. risk_status.confirming = true → AI 通过提示词自然确认（不弹窗）
  2. 用户重复高风险内容 → 自动确认 → Q9 兜底触发
  2b. AI SYS 中 risk_confirmed=true → Q9 兜底触发
  3. soothing_rounds（3 轮缓和）→ 恢复正常 interview
```

风险检测识别三类：`suicide_ideation`、`self_harm`、`extreme_despair`。所有风险处理通过自然对话完成，无弹窗。AI 风险确认阶段跳过内容过滤。

#### Q9 后兜底补问链

Q9 通过兜底提交完成后，系统自动检查是否有 pending 的非 Q9 条目。若有，跳过 3 轮缓和阶段，直接链式弹出下一个 pending 条目做兜底选择题。此机制确保高风险处理路径不"吞噬"其他条目的覆盖空间。



### 4.8 对话收尾（分级控制）

所有条目覆盖完毕、`is_done=true` 时，系统按风险等级注入差异化收尾语：

- **低风险（0-4）**：「今天聊到这儿。接下来，系统会为你生成一份个人报告，帮你把刚才聊到的状态整理清楚。」
- **有风险（5-14）**：「今天聊到这儿。谢谢你愿意把这些感受说给我听——有些话说出来本身就需要勇气。接下来系统会为你生成一份报告，帮你把一些模糊的感受看得更清晰。」
- **自杀意念**：「今天聊到这儿。你刚才说到的感受，我都听到了——你不是一个人面对这些。接下来系统会为你生成一份报告。有些力量你可能自己还没意识到，但它们已经在起作用了。之后如果需要，有人可以陪你一起往前走。」

收尾语由代码层固定文案注入，不由 LLM 生成。AI 回复以问号结尾时，收尾语追加在后面；否则用换行衔接。



### 4.9 对话中严格禁止


| 类别   | 禁止行为                                       |
| ---- | ------------------------------------------ |
| 结构暴露 | 暴露条目编号、量表名称、评分逻辑、条目总数                      |
| 问卷感  | 逐条提问、让学生感觉「在做问卷」、连续追问两个问题                  |
| 治疗干预 | 安慰（「一切都会好的」）、解释（「这是因为……」）、建议（「你应该……」）、过度共情 |
| 评估暴露 | 对话中给出任何评分或风险判断（「看来你的情况比较严重」等）              |
| 收束语  | 待覆盖条目不为空时说「聊到这」「今天我们聊了」「差不多了」「总的来说」等总结性话语  |
| 空洞共情 | 「我理解你」「这很正常」「很多人都这样」                       |

## 5. 评分引擎



### 5.1 两阶段评分架构（对标 AgentMental AGs→AGu）

**第一阶段 — 对话中逐项初评（AGs 角色）：**
每轮对话的 SYS 分析块中包含 `item_score_guess`（0-3/-1），AI 基于当前条目的单段对话给出即时推测。条目推进时，代码层将初评分写入 `session.item_scores_initial`，供第二阶段交叉校验。

**第二阶段 — 对话后全局校准（AGu 角色）：**
对话完成后，`POST /api/score` 读取完整对话记录，独立进行 PsyCoT 三步推理，并与第一阶段的初评分交叉校验。全局校准阶段可以看到全部 9 条目的对话上下文，发现并修正逐项初评可能存在的偏差。

**为什么两阶段？** 逐项初评时 LLM 只能看到当前条目对话，随后条目可能暗示对前条目的不同判断。全局校准阶段 LLM 看到全部 9 条目完整对话，可修正初评偏差。

### 5.2 PsyCoT 三步推理

**Step 1 — 症状锚定**：从对话中提取该条目的具体事实，引用学生原话中的关键词句。对话中完全没有该维度信息 → 标记"症状信息缺失"，`matched_anchor_index=0`，跳过后续步骤。

**Step 2 — 严重度判断**：综合频率、强度、影响范围。频率判断按照严格的锚点匹配规则：

| 锚点 | 频率词 |
|------|--------|
| **0**（完全没有） | 过去两周内完全未出现，或学生明确否认 |
| **1**（有几天） | 偶尔、有时、有几天、一两次、"不是每天都""不是每节都""不是每次都"；语境界定词（"经常但并非每天"→锚点1）；临界判定时宁低不高 |
| **2**（一半以上天数） | 经常（且未加限定词）、大多数日子、超过一半、一周三四天以上 |
| **3**（几乎每天） | 几乎每天、每天、天天、差不多天天 |

`severity_judgment` 中必须包含"过去两周内有多少天"的具体判断。

**Step 3 — 锚点匹配**：将 Step 2 的严重度判断与评分锚点对照，输出 `matched_anchor_index`（0/1/2/3）。`matched_anchor_index` **优先于** LLM 直接输出的 `score`，确保评分与 PHQ-9 标准锚点严格对齐。

#### Q8 情境归因排除规则

Q8 是最容易误判的条目——学生描述的"变慢"往往来自学业压力、疲劳、性格，而非临床精神运动改变：

| 场景 | 判定 | 锚点上限 |
|------|------|---------|
| 学业压力归因（考试、作业量） | 情境性反应 | 锚点 0–1 |
| 无外部归因 + 他人注意到 + 持续 | 临床信号 | 锚点 2–3 |
| 写作业慢（题难/不想做） | 伪信号，排除 | — |
| 收拾慢（性格/不着急） | 伪信号，排除 | — |

### 5.3 全局校准规则

1. 感知对话整体的情绪基调和功能状态，形成整体印象
2. 识别与整体印象明显矛盾的条目——偏离幅度 ≥ 2 分才需填写 `calibration_note` 说明修正原因
3. 1 分差异属正常浮动，不需校准
4. 条目之间相互独立——不得以"某维度分数高所以其他维度也该高"为由调整
5. 输出 `calibration_summary` 汇总校准情况

### 5.4 代码层硬规则

- **`answer_quality` 硬钳位**：`insufficient` 条目强制置 0（与 prompt 层双重保险）；`partial` 条目标记但不强制钳位
- **Q9 强制规则**：任何非零得分 → `risk_level` 强制 `severe`，硬编码于评分逻辑中
- **兜底得分直接使用**：`fallback_scores` 不参与语义判断，直接写入
- **`max_tokens=8192`**：容纳 PsyCoT 三步推理 + 全局校准完整 JSON 输出
- **评分结果缓存**：`session.score_result` 存在时直接返回，避免重复调用

### 5.5 风险分级阈值

| 总分      | 风险等级         | 学生报告用词 |
| ------- | ------------ | ------ |
| 0–4 分   | minimal（低风险） | 状态良好   |
| 5–9 分   | mild（轻度）     | 稍有起伏   |
| 10–14 分 | moderate（中度） | 需要关注   |
| 15–27 分 | severe（高度）   | 需要支持   |

**评分参考维度**（不只频率，综合判断）：


| 维度     | 说明                         |
| ------ | -------------------------- |
| 频率     | 多少天出现、是否持续、偶发还是规律          |
| 强度     | 用户描述的程度，如「有点」vs「非常」vs「完全」  |
| 影响范围   | 是否影响到学习、社交、日常生活功能          |
| 自我觉察程度 | 主动描述越详细，往往反映症状越有意识地存在      |
| 语气与语境  | 犹豫、否认、轻描淡写可能低估；直接、具体的描述更可信 |




### 5.3 PHQ-9 条目评分锚点及完整定义



#### Q1 · 兴趣/快感缺失（anhedonia）


| 原始条目 | 做事时提不起劲或没有乐趣                         |
| ---- | ------------------------------------ |
| 核心意图 | 过去两周，对平时感兴趣的事物的兴趣或愉悦感是否下降。重点在情感反应的变化 |
| 追问方向 | 是什么事 / 兴趣减退的范围有多广                    |



| 分数  | 锚点描述                 |
| --- | -------------------- |

| 原始条目 | 感到情绪低落、沮丧或绝望                               |

| 追问方向 | 低落的深度 / 有没有绝望感                             |

| --- | ------------------- |
| 0 分 | 情绪基本稳定，即使偶有波动也能自我调节 |
| 1 分 | 有几天低落，不持续，能自我调节     |
| 2 分 | 超过半数天低落或沮丧，调节困难     |
| 3 分 | 几乎每天持续低落或绝望，感觉无法改变  |




#### Q3 · 睡眠紊乱（sleep）


| 原始条目 | 入睡困难、睡眠不稳或睡眠过多                           |
| ---- | ---------------------------------------- |
| 核心意图 | 过去两周，睡眠是否出现异常，包括失眠和睡眠过多两个方向。主观感受优先于小时数判断 |
| 追问方向 | 睡多还是睡少（方向先于频率）                           |



| 分数  | 锚点描述              |
| --- | ----------------- |
| 0 分 | 睡眠正常              |
| 1 分 | 偶尔睡不好或睡太多，不影响日常   |
| 2 分 | 睡眠问题较频繁，影响第二天状态   |
| 3 分 | 几乎每天严重睡眠问题，严重影响功能 |




#### Q4 · 精力/疲惫（energy）


| 原始条目 | 感到疲倦或没有活力                                |
| ---- | ---------------------------------------- |
| 核心意图 | 过去两周，是否经常感到疲劳或缺乏完成日常事务的精力。需与体力劳动后的正常疲劳区分 |
| 追问方向 | 程度 / 对日常生活的影响                            |



| 分数  | 锚点描述               |
| --- | ------------------ |
| 0 分 | 精力正常，适当休息后能恢复      |
| 1 分 | 有几天比平时累，但能完成日常     |

| 3 分 | 几乎每天极度疲劳，连基本事务也很费力 |




#### Q5 · 食欲变化（appetite）


| 原始条目 | 食欲不振或吃太多                                        |
| ---- | ----------------------------------------------- |
| 核心意图 | 过去两周，食欲是否出现明显变化，包括减退和暴食/情绪性进食两个方向。重点是与个人平时基线的变化 |
| 追问方向 | 吃多还是吃少（方向先于频率）                                  |



| 分数  | 锚点描述              |
| --- | ----------------- |
| 0 分 | 食欲正常              |

#### Q6 · 自我评价/自责（self-worth）

| ---- | ------------------------------------------------ |

| --- | ---------------------- |
| 0 分 | 对自己基本中立或正向，即使犯错也不会过度自责 |
| 1 分 | 偶尔觉得自己不够好，短暂           |
| 2 分 | 经常觉得自己很差劲，影响自我感受       |
| 3 分 | 几乎每天强烈自我否定或觉得自己是负担     |




#### Q7 · 注意力（concentration）


| 原始条目 | 对事务专注有困难，例如阅读报纸或看电视                    |
| ---- | -------------------------------------- |
| 核心意图 | 过去两周，集中注意力完成任务的能力是否下降。对学生群体尤为关键，直接影响学业 |
| 追问方向 | 影响到哪些具体事情（学习/日常）                       |



| 分数  | 锚点描述            |
| --- | --------------- |
| 0 分 | 注意力正常           |
| 1 分 | 偶尔难以集中，轻微       |
| 2 分 | 经常难以集中，影响学习效率   |
| 3 分 | 几乎无法集中，严重影响日常功能 |




#### Q8 · 精神运动变化（psychomotor）


| 原始条目 | 行动或说话缓慢以至于别人能注意到，或者烦躁不安、动来动去、坐立不安               |
| ---- | ----------------------------------------------- |
| 核心意图 | 过去两周，是否出现行动/语言迟缓或激越，且程度明显到他人可察觉。包括精神运动迟滞和激越两个方向 |
| 追问方向 | 自己感觉到还是别人注意到                                    |



| 分数  | 锚点描述          |
| --- | ------------- |
| 0 分 | 与平时无异         |
| 1 分 | 偶尔轻微变化，本人有感觉  |
| 2 分 | 经常明显，本人觉察明显   |
| 3 分 | 几乎每天明显，他人也注意到 |




#### Q9 · 自伤/自杀意念（self-harm）⚠️ 高风险条目


| 原始条目 | 有不如死掉或用某种方式伤害自己的想法                           |
| ---- | -------------------------------------------- |
| 核心意图 | 过去两周，是否出现任何形式的自伤念头、被动死亡意念或主动自杀想法。是风险分级的决定性条目 |
| 追问方向 | **仅确认有意念，不多追问**                              |



| 分数  | 锚点描述                              |
| --- | --------------------------------- |
| 0 分 | 完全没有这类想法                          |
| 1 分 | 偶有被动死亡意念（「消失就好了」「希望睡过去不醒来」），无主动想法 |
| 2 分 | 有主动想法但无具体计划                       |
| 3 分 | 有具体计划或行为                          |


**特殊约束**：主问题以非直接方式引出，降低防御性。不在前三轮触发。用户说没有则立即接受。任何非零得分立即触发安全流程。

### 5.4 风险分级


| 总分      | 风险等级         | 学生报告用词（去病理化） | 报告建议方向                 |
| ------- | ------------ | ------------ | ---------------------- |
| 0–4 分   | minimal（低风险） | 状态良好         | 自我关怀建议为主，鼓励性语气         |
| 5–9 分   | mild（轻度）     | 稍有起伏         | 自我关怀 + 自我调节建议，温和关注     |
| 10–14 分 | moderate（中度） | 需要关注         | 自我关怀 + 社交支持 + 校内心理咨询提示 |
| 15–27 分 | severe（高度）   | 需要支持         | 自我关怀 + 专业转介建议 + 危机热线   |


**Q9 特殊规则**：

- Q9 任何非零得分 → **风险等级强制设为 severe**，不受总分影响
- `q9_nonzero` 标记为 true，触发学生报告和教师报告的安全模块

## 6. 报告引擎



### 6.1 设计哲学

报告的设计目标是让用户感受到：**被看见、被理解、被赋能**——看见自己已有的力量，从中获得继续向前走的信心。

- **不展示分数和等级标签**。学生版报告不使用「你的得分」「你属于 XX 级别」等表述。
- **去病理化**。不使用「症状」「抑郁」「障碍」「风险」等词汇。
- **锚定对话原文**。每一条观察和解释必须有对话中的具体内容作为依据；信息不足时使用兜底表述。
- **从分数到洞察**。系统不告诉学生「你得了多少分」，而是帮助他发现：自己已经拥有了哪些应对生活的力量。



### 6.2 学生版报告结构

学生报告由四个模块构成，严格按顺序呈现：

```
┌─────────────────────────────────────┐
│  ① 状态凝练句                        │
│  ← 10-15 字，从用户原话提炼，大字居中   │
├─────────────────────────────────────┤
│  ② 此刻的你                          │
│  第一层 · 看见（1 句）                 │
│  第二层 · 理解（3-4 句）               │
│  第三层 · 力量（2 句）                 │
├─────────────────────────────────────┤
│  ③ 往前一步                          │
│  引导语 + 建议 3-5 条                  │
│  [Q9 非零时：危机支持提示]             │
├─────────────────────────────────────┤
│  ④ 底部：重新开始                      │
└─────────────────────────────────────┘
```



#### 模块一：状态凝练句

报告顶部展示一句由 AI 从用户对话中凝练出的话，作为整份报告最显眼的视觉元素。凝练句不采用单次生成，而是走**「两选一 + 自检」**流程以保证输出质量：

1. **LLM 同时生成两个候选凝练句**，从不同角度切入（一个偏「承认处境，但不被处境定义」，一个偏「锚定用户身上已有的力量」）
2. **LLM 对每个候选做二元自检**——「这个句子是否满足：来自用户原话、落点在力量或韧性、不诗意不鸡汤、不引入用户没说过的概念？」
3. **择优展示**：两个都通过→随机选用一个；只有一个通过→展示那个；两个都不通过→使用固定兜底模板


| 属性  | 要求                             |
| --- | ------------------------------ |
| 来源  | LLM 从用户对话中提炼，所有候选必须锚定用户原话      |
| 长度  | 10–15 字，一句话，不折行                |
| 语气  | 保留用户原有的措辞风格和语气                 |
| 视角  | 描述性而非诊断性——说的是用户的感受和处境，不是对用户的评判 |
| 落点  | 落在用户身上存在的力量、清醒或韧性——承认困难，但有出路感  |
| 禁止  | 病理化词汇、散文化或诗意化表达                |


**兜底模板**（两个候选均未通过自检时使用）：

> 「完成了一次认真的对话，对自己的了解又多了一点。」



#### 模块二：此刻的你（三层递进）


| 层次       | 定位               | 篇幅    | 要求                |
| -------- | ---------------- | ----- | ----------------- |
| 第一层 · 看见 | 说出用户身上正在发生什么     | 1 句   | 不列举清单，不说教         |
| 第二层 · 理解 | 说明这些感受和状态背后的心理逻辑 | 3-4 句 | 日常语言，不推断童年经历/家庭模式 |
| 第三层 · 力量 | 发现用户已经在发挥的力量     | 2 句   | 来自真实对话内容，用事实说话    |




#### 模块三：往前一步（差异化建议）

按「发现你的状态→理解你的需要→匹配你的方向」逻辑展开，每条 30-40 字。


| 风险等级            | 引导语基调 | 建议方向                                      |
| --------------- | ----- | ----------------------------------------- |
| minimal（0-4）    | 轻松肯定  | 保持喜欢的小事、规律作息、保持社交连接                       |
| mild（5-9）       | 温和支持  | 自我关怀 + 情绪记录、身体活动、设定边界等                    |
| moderate（10-14） | 关切有力  | 自我关怀 + 社交支持 + 「可以考虑和学校心理老师聊聊」             |
| severe（15-27）   | 稳定清晰  | 具体自我支撑方法 + 「建议寻求专业支持」 + 危机热线 400-161-9995 |


**Q9 非零时的特殊处理**：建议模块最前端自然融入固定安全话术，不由 AI 生成：

> 「你提到了有时候会有一种想消失的感觉——这样的感受是真实的，值得被认真对待。如果你需要和人说说话，全国心理援助热线 400-161-9995 随时可以打。」



#### 报告严禁

- 展示分数、等级标签、风险程度
- 病理化词汇（「抑郁」「焦虑症状」「障碍」）
- 散文化、诗意化、治愈系公众号语气
- 模板化、通用化表述
- 推断童年经历、家庭模式、依恋模式、创伤等
- 「你其实……」「你真正……」「你内心深处……」等强行解读表述



### 6.3 教师版报告结构



#### 层次一：个案报告


| 模块      | 内容                                           |
| ------- | -------------------------------------------- |
| 基本信息    | 学生标识、筛查日期、完成状态                               |
| 整体评估    | 总分、风险等级，与上次筛查对比（如有历史数据）                      |
| Q9 安全标注 | Q9 非零时醒目提示，附带对话原文摘录，标明被动意念/主动念头/有无计划         |
| 高分条目解析  | ≥2 分的条目逐一列出，附原因分析和对话片段摘录                     |
| 对话摘要    | AI 生成的客观对话摘要，保留学生原话中的关键表达方式                  |
| 建议跟进行动  | 按风险等级给出的教师行动建议：日常关注/约谈建议/通知家长建议/转介心理老师/紧急干预等 |




#### 层次二：班级/年级汇总报告


| 模块      | 内容                            |
| ------- | ----------------------------- |
| 筛查概览    | 本次应筛人数、实际完成人数、完成率、各风险等级人数及百分比 |
| 维度热力图   | 9 个维度的班级平均得分矩阵，颜色深浅表示得分高低     |
| Q9 安全汇总 | Q9 非零学生数量单独标注，列出名单（脱敏或按权限）    |
| 变化趋势    | 与上次筛查对比的变化方向（需历史数据支持，P2 功能）   |
| 关注信号摘要  | AI 生成的群体层面关注点摘要               |

## 7. 安全与风险管理

安全机制是 COSMO 最重要的非功能需求。系统从对话、评分、报告三个阶段对风险进行层层拦截。

### 7.1 双层安全防护

系统从对话和评分两个阶段对风险进行层层拦截，报告阶段嵌入固定安全模块。

```
第一层 · 对话阶段 ──→ 风险检测（关键词+正则）→ 自然对话确认（无弹窗）→ Q9 兜底 + 3 轮缓和 → 后处理过滤（病理化词汇）
第二层 · 评分阶段 ──→ Q9 强制规则（非零→severe）
第三层 · 报告阶段 ──→ 危机热线固定模块（不可绕过）
```



### 7.2 第一层：对话阶段 — 双层防护

#### 防护一：风险检测（前置）

在对话过程中实时识别用户消息中的三类风险：自杀意念（`suicide_ideation`）、自伤（`self_harm`）、极度绝望（`extreme_despair`）。检测方式：关键词 + 正则匹配（`detectRisk`），否定语境快速出口避免误匹配。

**响应流程**：

1. 首次检测到风险 → `risk_status.confirming = true` → AI 通过 prompt 自然确认用户意图（不弹窗）
2. 用户再次高风险内容 → 自动确认 → 触发 Q9 兜底 + `soothing_rounds = 3`
3. AI SYS 中 `risk_confirmed = true` → 也触发 Q9 兜底
4. 3 轮缓和后 → 恢复正常 interview

#### 防护二：后处理过滤（后置）

`post-processor.ts` — AI 回复生成后，逐句过滤 5 条病理化硬底线词汇（抑郁/焦虑/障碍/症状/诊断）。含违禁词的句子整句移除，全部被过滤时使用安全兜底文案。风险确认阶段跳过过滤。

### 7.3 第二层：评分阶段 Q9 强制规则

- Q9 任何非零得分 → risk_level 强制设为 severe
- q9_nonzero 标记为 true，不受其他条目得分影响
- 此规则**硬编码于评分逻辑中**，不依赖模型输出，无法被绕过



### 7.4 第三层：报告阶段安全模块


| 对象     | 安全模块内容                             |
| ------ | ---------------------------------- |
| 学生报告   | 建议模块最前端插入固定危机热线文案，不由 AI 生成，不可被省略   |
| 教师个案报告 | 顶部醒目位置标注高风险：有自杀/自伤意念，附带对话摘录，标明意念性质 |
| 教师汇总报告 | 高风险学生数量单独列出，触发班主任个案跟进提醒            |




### 7.5 危机热线与资源

- 全国心理援助热线：**400-161-9995**（24 小时）
- 希望 24 热线：400-161-9995
- 校内心理老师联系方式（由学校管理员配置）

## 8. 引擎可复用性设计

COSMO 引擎与量表解耦。PHQ-9 是当前的示范量表——换量表只需替换配置，引擎核心不变。

### 8.1 量表配置文件

每个量表对应一个独立配置文件（`src/lib/scales/*.ts`），包含：

- 量表名称和描述（用于开场透明告知文案）
- 条目列表：原始条目文本 + core_intent + 追问方向 + is_high_risk 标记 + fallback_question + fallback_options（四个，对应 0-3 分）+ score_anchors（四个，对应 0-3 分的语义锚点）
- 风险分级阈值（minimal / mild / moderate / severe 的分值范围）
- 高风险条目的报告融入话术（如适用）



### 8.2 引擎不变的部分（换量表无需改动）


| 模块   | 不变的内容                                |
| ---- | ------------------------------------ |
| 开场引擎 | 透明告知框架、破冰对话逻辑                        |
| 对话引擎 | 单次回应结构（共鸣·追问决策）、条目覆盖追踪机制、快捷短语、兜底评分机制 |
| 评分引擎 | PsyCoT 三步推理（Step 1-3）+ 全局校准、评分参考维度            |
| 报告引擎 | 学生报告三层结构（凝练句·状态分析·有方向）、教师报告结构        |
| 安全系统 | 三层安全防护、高风险实时检测、Q9 强制规则               |
| UI 层 | 对话页结构、报告页布局、动效规范                     |




### 8.3 接入新量表所需改动

- 新增量表配置文件（条目原文、core_intent、追问方向、高风险标记、fallback 配置）
- 更新开场透明告知中的量表名称和描述文字
- 更新高风险条目的报告融入话术（如适用）
- 调整风险分级阈值（如需）

UI、对话逻辑框架、报告结构、System Prompt 主体均无需改动。

## 9. UI 设计系统



### 9.1 设计方向：Light Mode 校园访谈氛围

V3.2 将视觉体系全面升级为 Light Mode（校园访谈氛围）。

**设计目标**：

- **安全感**——色调温暖、不压迫，让学生感到安心而非紧张
- **去病理化**——不传递任何「医院」「诊所」「测试」的视觉暗示
- **非审判**——克制、不喧哗，像一间安静谈话的教室
- **校园环境适配**——明亮但不刺眼，适合日常教学环境中使用



### 9.2 配色规范


| 角色     | 色值             | 用途               |
| ------ | -------------- | ---------------- |
| 背景色    | #F7F6F2（米白）    | 全局页面背景，温和不刺眼     |
| 主色     | #A3B5A0（浅鼠尾草绿） | 按钮、强调元素、进度指示     |
| 文字主色   | #2D2D2D（深灰）    | 正文、标题            |
| 文字辅色   | #787878（中灰）    | 说明文字、占位文本        |
| AI 气泡  | #F2F0ED（米灰）    | 接近访谈记录纸的质感，左对齐   |
| 用户气泡   | #E8EEF2（淡蓝灰）   | 与 AI 气泡有明显区分，右对齐 |
| 分隔线/边框 | #E5E2DE        | 卡片边框、分隔线         |
| 积极指示   | #7BAE7F        | 低风险标签、正面反馈       |
| 关注指示   | #E8B44F（柔和琥珀）  | 中度关注标签           |
| 严重指示   | #C26A5A（柔和暗红）  | 高度关注标签（非纯红，降低刺激） |


**配色禁用**：

- 深色背景
- 强对比配色（纯黑 #000 + 纯白 #FFF）
- 纯红色警示（#FF0000 等）

### 9.3 页面设计要点



#### 开始页

- 居中布局，克制文字。产品名 + 一行说明 + 开始按钮

- 三个小图标（安全保密/自然对话/了解自己）+ 短标签，建立信任感



#### 对话页


| 元素    | 设计要点                                            |
| ----- | ----------------------------------------------- |
| 聊天气泡  | 轻量、接近访谈记录纸质感。AI 气泡左对齐（#F2F0ED），用户气泡右对齐（#E8EEF2） |
| 进度指示  | 极简：只显示大致阶段（「开始中」「快结束了」），不暴露条目数和编号               |
| 输入区   | 简洁：输入框（圆角 16px）+ 发送按钮 + 快捷短语小按钮。固定于页面底部         |

- 右上角「查看报告」按钮在对话全程始终可见



#### 学生报告页


| 元素   | 设计要点                     |
| ---- | ------------------------ |
| 凝练句  | 10-15 字大字居中显示，充足留白，截图友好  |
| 分析段落 | 行距宽松，段间留白明显              |
| 建议列表 | 每条独立呈现为卡片或分隔段落           |
| 危机提示 | 融合在建议开头，柔和暖色背景轻微区分，不用警告红 |




### 9.4 动效原则

- 所有过渡使用 ease-out 曲线，时长 300–500ms
- 气泡以轻微上浮 + 淡入方式出现（y: 10px, opacity: 0 → 1）
- 只做必要的功能性动效，不做装饰性动效
- 动效节奏宁慢不偏快，保持沉静感
- 不做快速弹出、跳切、缩放等刺激感动效



### 9.5 响应式要求

- 学生端：优先适配手机（375-414px 宽度）
- 教师端：适配桌面（1024px+）
- 聊天气泡在手机上最大宽度 ≤ 80%

## 10. 技术架构



### 10.1 技术选型


| 模块       | 技术方案                        | 说明                         |
| -------- | --------------------------- | -------------------------- |
| 框架       | Next.js 14（App Router）      | 纯客户端页面 + API Routes，服务端渲染 |
| 语言       | TypeScript                  | 全栈类型安全                     |
| 样式       | Tailwind CSS 3.4            | Utility-first               |
| LLM 服务   | DeepSeek API（通过 OpenAI SDK 兼容接入） | 对话模型 `deepseek-v4-pro`，评分/报告模型默认同 |
| LLM 调用 | `textCompletion`（文本对话）/ `jsonCompletion`（JSON 模式）/ `createRealStreamResponse`（流式） | 服务端调用，API Key 不暴露给前端 |
| 会话存储 | 内存 Map（`globalThis.__cosmo_sessions`） | 无数据库，服务重启丢失 |
| 量表配置 | `src/lib/scales/phq9.ts` | 每个量表独立配置文件，与引擎解耦 |
| 部署 | Vercel / 自托管 | 支持 Edge Functions |
| API Key | 环境变量 `DEEPSEEK_API_KEY` | 服务端持有，前端不可见 |




### 10.2 API Route 设计


| Route              | 功能     | 输入                         | 输出                             | 关键逻辑              |
| ------------------ | ------ | -------------------------- | ------------------------------ | ----------------- |
| `POST /api/session`   | 会话管理  | — | session_id                      | 创建新会话、初始化 SessionState |
| `POST /api/chat`   | 对话主接口  | session_id + user_message | AI 流式回复 + 元数据（`__META__`分隔符）     | SYS 解析 + 四分支决策 + 风险检测 + 后处理过滤 + 覆盖追踪 |
| `POST /api/score`  | 评分接口   | session_id | 各条目分数 + 总分 + risk_level + calibration_summary | PsyCoT 三步推理 + 全局校准 + Q9 强制规则 + 信息不足硬钳位 |
| `POST /api/report` | 报告生成接口 | session_id | 凝练句 + 状态分析 + 建议列表              | 三层报告结构 + 病理化过滤 |




### 10.3 数据流

```
浏览器 (CSR)
  ├── /                  首页 → 创建会话 → 跳转 /chat
  ├── /chat              对话页 → 流式接收 AI 回复
  └── /report            报告页 → 依次调用评分 API、报告 API
                              ↑
POST /api/session  ─── 创建会话，返回 session_id
POST /api/chat     ─── 核心对话逻辑（流式响应，__META__ 分隔符）
POST /api/score    ─── PHQ-9 语义评分（PsyCoT + 全局校准）
POST /api/report   ─── 生成个性化报告（JSON 模式）
```

流式响应协议：

```
AI回复内容逐块流式输出...\n__META__\n{"show_fallback":false,"fallback_item":null,"is_done":false,...}
```

前端 `chat/page.tsx` 通过 `__META__` 分隔符分割流式内容和结构化元数据。

### 10.4 Session 状态管理

Demo 阶段使用服务端内存 Map（`globalThis.__cosmo_sessions`）持有 SessionState。服务重启后数据丢失。核心状态字段：phase、coverage、item_rounds、item_scores_initial、score_result、report_result 等。

### 10.5 缓存策略

- **评分结果缓存**：`session.score_result` 存在时直接返回，避免重复调用评分 API
- **报告结果缓存**：`session.report_result` 存在时直接返回
- **前端报告页缓存**：`app/report/page.tsx` 模块级 `reportCache`（Map），同一 session 二次进入直接读取

**关键指标**：
- 对话完成率：100%（4/4 全部自然完成）
- 覆盖率：100%（4/4 案例 9 条目全部覆盖）
- 评分准确率（±2分以内）：50%（案例3 完美匹配、案例4 仅差1分）

**已知问题**：
1. 🔴 **评分系统性高估**：案例2（+6），Q4（精力）、Q7（注意力）为高估重灾区
2. 🟡 **SYS 块输出率偏低**：5%~28%，多数轮次依赖代码层 fallback
3. 🟡 **Q9 硬编码规则误判**：案例2/3 中 Q9=1 触发 severe，实际临床更接近 mild/moderate
4. 🟡 **边缘判断一致性**：情境性症状判定标准不统一

### B. 关键学术参考

- **HopeBot (UCL/NHS, 2025)**：AI 对话 PHQ-9 评分与自评 ICC = 0.92，71% 更信任机器人评分
- **AgentMental (AAAI, 2026)**：四 Agent 协作，自适应追问贡献约 24% Kappa，结构化记忆贡献约 26% Kappa
- **MAGI PsyCoT**：三步思维链推理（症状锚定→严重度判断→锚点匹配）
- **BDI-FS-GPT (JMIR, 2026)**：AI 对话评分的临床一致性优于自评（κ = 0.72 vs 0.55）
- **Perla (Madrid, 2020)**：AI 筛查触达率是传统表单的 2.5 倍



### C. 风险等级去病理化表述对照


| 专业等级            | 学生报告用词 | 含义                        |
| --------------- | ------ | ------------------------- |
| minimal（0-4）    | 状态良好   | 情绪状态良好，目前没有需要特别关注的信号      |
| mild（5-9）       | 稍有起伏   | 最近情绪有些起伏，可能需要更多关注自己的状态    |
| moderate（10-14） | 需要关注   | 你在承受比较大的心理压力，建议认真对待并寻求支持  |
| severe（15-27）   | 需要支持   | 你目前的状态需要专业的支持和帮助，请不要一个人扛着 |




### D. 术语表


| 术语          | 英文/缩写                          | 定义                                             |
| ----------- | ------------------------------ | ---------------------------------------------- |
| PHQ-9       | Patient Health Questionnaire-9 | 九条目患者健康问卷，用于抑郁症状筛查                             |
| 构念一致性       | Construct Validity             | AI 改写后的提问是否真实覆盖了量表条目原本要测量的心理学构念                |
| 锚点          | Anchors                        | 每个分值对应的语义描述，是评分匹配的参照标准                         |
| 语义聚合        | Semantic Aggregation           | 将多轮分散的用户回答整合为完整的待评分语义片段                        |
| 兜底机制        | Fallback                       | 当自然对话无法获取足够评分信息时，用四选一按钮收集所需信息                  |
| QLV         | Question Validation Layer      | 问题校验层，校验 AI 提问是否有效覆盖核心意图。V3.5 中已由 SYS 分析块 + 代码层决策替代 |
| core_intent | 核心意图                           | 量表条目的核心测量目标，AI 提问不得偏离或改变此含义                    |
| risk_level  | 风险等级                           | minimal / mild / moderate / severe 四级，决定报告建议策略 |
