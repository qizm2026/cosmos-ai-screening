## 1. 产品概述

### 1.1 产品定义

COSMO 是一款面向学校心理安全管理体系的 AI 半结构化心理访谈辅助工具。以 PHQ-9 为当前示范量表，通过**自然对话完成心理筛查**——将「测量」伪装成「聊天」，让学生在放松、非防御的状态下自然表达，同时帮助教师在专业框架下获取更丰富的评估信息。

### 1.2 产品定位与使用场景


| 维度   | 说明                                                   |
| ---- | ---------------------------------------------------- |
| 产品形态 | B2B SaaS，售卖对象为教育局，部署于学校心理预警平台内                       |
| 核心场景 | 学校完成全体学生大规模初筛后，班主任对预警学生进行二次核查时使用，AI 作为半结构化心理访谈辅助工具介入 |
| 设计目标 | 支持教师判断，而非替代判断                                        |
| 专业价值 | 提升专业性与构念一致性，降低误判与情绪引导偏差风险                            |
| 业务闭环 | 打通「初筛→复核→报告→跟进」的学校心理安全管理闭环                           |




### 1.3 相比传统量表的三个核心优势


| 维度   | 传统量表                   | COSMO                         |
| ---- | ---------------------- | ----------------------------- |
| 即时反馈 | 学生做完无反馈，只能等老师通知        | 对话结束后即时生成自我觉察报告，帮助学生了解自己      |
| 作答质量 | 条目固定、机械重复，易引发练习效应和防御心理 | AI 动态改写条目为邀请式自然对话，降低防御，利于真实作答 |
| 信息深度 | 只能获取 0-3 的离散分值，无背景和细节  | 自然语言回答可提取临床有效信息，支持更准确的评分判断    |




### 1.4 定位边界

- 定位为**筛查辅助工具**，帮助教师判断，不替代临床诊断
- AI 作为**半结构化访谈辅助**，而非自由心理咨询师
- 输出**评估参考**，而非诊断结论
- 所有**安全相关决策权归属教师和专业心理人员**



### 1.5 当前示范量表

PHQ-9（Patient Health Questionnaire-9），用于抑郁症状筛查。引擎设计为**量表无关**，更换配置即可接入其他量表（如 GAD-7 焦虑筛查等）。

### 1.6 核心设计原则


| 原则       | 说明                                                                |
| -------- | ----------------------------------------------------------------- |
| **专业优先** | 筛查不是娱乐。AI 提问必须覆盖量表条目的核心构念（core_intent），不得改变心理学构念含义，不得引入额外变量或扩展解释。 |
| **对话自然** | 把「测量」伪装成「聊天」。学生感觉在被倾听而非被评估，对话节奏由用户掌控，不逐条提问。                       |
| **去病理化** | 全程不使用「症状」「抑郁」「障碍」「诊断」「患者」等标签化词汇。                                  |
| **安全第一** | 自杀/自伤条目的特殊规则，对话阶段即嵌入实时风险检测，Q9 任何非零得分强制触发安全流程。                     |
| **知行分离** | 对话中严禁安慰、解释、建议或任何治疗性干预——只倾听与记录。建议只在报告里给。                           |
| **透明信任** | 在对话开始前坦诚告知：这是什么、能得到什么、数据怎么处理。信任是真实作答的前提。                          |
| **可复用**  | 引擎与量表解耦，PHQ-9 只是参数。换量表只需替换配置文件，引擎主体无需改动。                          |

## 2. 用户旅程



### 2.1 学生端全流程

学生完成一次筛查的完整路径。整体体验目标控制在 **8–12 分钟**。


| 步骤     | 页面  | 核心内容                         | 时长      | 要点                 |
| ------ | --- | ---------------------------- | ------- | ------------------ |
| Step 1 | 开始页 | 透明告知（是什么/能得到什么/隐私怎么处理）+ 自然破冰 | 1–2 min | 不设「确认开始」按钮，破冰后自然过渡 |
| Step 2 | 对话页 | 自然访谈式对话，动态覆盖 PHQ-9 全部 9 个条目  | 5–8 min | 高风险实时检测就绪          |
| Step 3 | 承接页 | 「谢谢你和我聊天」+「查看报告」按钮           | 短暂停留    | 用户主动点击，不自动跳转       |
| Step 4 | 报告页 | 凝练句 + 三层递进状态分析 + 差异化建议       | 阅读为主    | 三层递进：观察→解释→资源      |




#### Step 1 · 开始页

核心目标是用透明度建立信任。学生在进入对话前，以轻松、不压迫的方式了解三件事：

- **这是什么**：一次关于你近期状态的对话探索，帮你了解自己最近的情绪和精力。
- **你会收获什么**：一份基于这次对话的个性化报告，包含对你状态的描述和一些参考建议。
- **隐私说明**：你的对话内容仅用于生成本次报告，不会被用于其他用途。



#### Step 3 · 承接页

在对话结束和报告展示之间提供心理缓冲。不自动跳转，由学生主动点击按钮进入报告。

### 2.2 教师端全流程

教师端面向班主任和心理老师，提供两个层次的数据视图：

#### 层次一：班级/年级汇总报告


| 模块      | 内容                                     |
| ------- | -------------------------------------- |
| 筛查概览    | 本次筛查人数、完成率、各风险等级人数及占比                  |
| 维度热力图   | 9 个维度的班级平均分矩阵，识别群体层面得分偏高的维度，辅助制定团体干预方向 |
| Q9 安全汇总 | Q9 非零学生数量单独标注，触发个案跟进提醒，名单可供班主任导出       |
| 变化趋势    | 与上次筛查对比的变化方向（需历史数据支持，P2 功能）            |
| 关注信号摘要  | 整体层面的关注信号，如高关注等级学生数量、本次新增预警数量等         |




#### 层次二：个案报告


| 模块      | 内容                             |
| ------- | ------------------------------ |
| 基本信息    | 学生标识、筛查日期、完成状态                 |
| 整体评估    | 总分、风险等级、与上次筛查对比（如有历史数据）        |
| Q9 安全标注 | Q9 非零时醒目提示，附带对话原文摘录，供教师了解严重程度  |
| 逐项条目解析  | 9 个条目逐一列出，附最终评分、分析理由和对话摘录      |
| 对话摘要    | AI 生成的客观摘要，保留学生原话中的关键表达        |
| 建议跟进行动  | 按风险等级给出的教师行动建议（日常关注/约谈建议/转介提示） |

## 3. 开场引擎

开场的目标是用透明度建立信任。开场不设单独的「进入确认」步骤，由破冰对话自然过渡到第一个正式条目。

### 3.1 透明告知模块


| 告知项    | 内容                               |
| ------ | -------------------------------- |
| 这是什么   | 一次关于你近期状态的对话探索，帮你了解自己最近的情绪和精力。   |
| 你会收获什么 | 一份基于这次对话的个性化报告，包含对你状态的描述和一些参考建议。 |
| 隐私怎么处理 | 你的对话内容仅用于生成本次报告，不会被用于其他用途。       |
| 需要多久   | 大约 8–10 分钟，可以随时暂停退出。             |




### 3.2 破冰对话

用户点击开始后，AI 直接进入对话，不使用「在我们开始之前……」之类制造割裂感的引导语。破冰问题应是真正轻松的日常话题，与测评条目内容明显区分，目的是建立关系感，降低防御性。

**破冰问题设计原则**：

- 轻松、开放，与情绪健康评估完全无关
- 让学生有话可说，不会答错，不会引发自我审视
- AI 对回答的回应要真实共鸣，不能用通用话术敷衍
- 破冰不超过 2 轮，之后 AI 自然收尾，顺势进入第一个正式条目

**破冰问题示例（符合原则）**：

- 「最近有没有什么让你觉得有点意思的事，哪怕很小的事情也算？」
- 「最近有在追什么剧或者玩什么吗，还是忙到没时间？」

**破冰问题反例（避免）**：

- 「最近生活节奏怎么样？是那种忙起来停不下来的感觉，还是反而有点太闲？」——触及精力/空虚感，与 Q4/Q1 内容重叠，会提前引发测评感。

**自然过渡示例**：
破冰第 2 轮结束后，AI 直接说：「好，那聊聊最近的状态吧——」然后进入第一个正式条目，不设确认按钮。

## 4. 对话引擎

对话引擎是 COSMO 的核心驱动模块。它负责从开场破冰到条目覆盖、从追问决策到自然收尾的完整对话流程。

### 4.1 引擎设计原则


| 原则          | 说明                                                                     |
| ----------- | ---------------------------------------------------------------------- |
| **自然访谈风格**  | AI 是一个专业、有温度的对话者，不是问卷机器。每次回应都在倾听和承接的基础上展开。                             |
| **条目覆盖优先**  | 量表所有条目最终必须全部被覆盖，顺序跟随对话自然调整，不按固定顺序。                                     |
| **用户掌控节奏**  | 支持快捷短语，支持自由输入，AI 不施压、不催促。                                              |
| **严禁治疗性干预** | 对话中不安慰、不解释、不建议、不过度共情——这些属于报告和建议的范畴。但同时不能忽视用户传达出的消极情绪，在必要时需给予轻量教育指引或辅导。 |
| **严禁泄露结构**  | 不暴露条目编号、量表名称、评分逻辑。学生不知道自己在「做问卷」。                                       |
| **严禁即时评估**  | 对话中不给出任何评分或风险判断，不暗示好坏对错。                                               |




### 4.2 对话角色设定

AI 扮演一位温暖、专业的学校心理老师，在轻松的课间聊天氛围中完成筛查。

- 称呼学生为「你」，不叫「用户」「测试者」
- 使用贴近青少年的口语化表达，站在一种同辈/朋友的视角平等交流
- 保持专业性的同时展现适度的亲和度
- 学生全程不知道有 9 个条目、不知道评分标准
- 对话包含：针对量表题目的**正式提问**、**聊天式追问**、**自然承接的聊天式对话**



### 4.3 正式提问生成规则

AI 在正式提问前，需根据以下三个输入动态生成问题：

1. 量表原始条目文本
2. 该条目的核心意图（core_intent）
3. 当前对话上下文

改写不使用固定模板，由模型基于上下文灵活生成。

#### 4.3.1 构念一致性（强约束）

所有条目的 AI 正式提问必须严格满足：

- 必须覆盖量表条目的 core_intent，不得改变心理学构念含义
- 不得引入额外变量或扩展解释
- 必须体现至少以下维度中的两项：**行为（behavior）、主观体验（feeling）、时间/频率（temporal）**



#### 4.3.2 改写输出原则


| 原则          | 说明                  | 示例                                |
| ----------- | ------------------- | --------------------------------- |
| 陈述句→开放式疑问句  | 不直接定义用户状态，邀请用户描述体验  | 「做事提不起劲」→「最近做喜欢的事时，感受和以前有什么不一样吗？」 |
| 抽象概念→具体生活情境 | 学生不理解心理学术语，但理解自己的生活 | 「你是否感到沮丧？」→「过去两周，有没有哪几天感觉特别低落？」   |
| 避免标签感/评判感   | 不使用「失败」「问题」「异常」等词   | 「我是失败者」→「会不会有时候觉得自己不够好？」          |
| 安全优先        | Q9 以非直接方式引出，降低防御性   | 见高风险条目特殊处理                        |




#### 4.3.3 风格禁止清单

- 隐喻/比喻表达（如「像被困住」「心里有个黑洞」）
- 文艺化表达（避免散文式、诗意化语言）
- 情绪过度渲染（保持克制，不煽情）
- 选择题结构（如「是 A 还是 B？」）



#### 4.3.4 语气要求


| 维度   | 倾向              |
| ---- | --------------- |
| 专业性  | 高               |
| 亲和度  | 中等              |
| 情绪引导 | 弱（不进行心理解释或诊断暗示） |




### 4.4 单次回应结构与追问决策

AI 每次回应分两步执行：

**Step 1 · 共鸣回应**：自然承接用户的对话，像与朋友聊天一样，可以适当使用咨询技巧，如：解释、澄清、共情等。

- ✅ 自然、具体的共鸣，锚定用户刚才说的内容
- ❌ 「我理解你」「这很正常」「很多人都这样」——**空洞共情，严禁使用**

**Step 2 · 追问决策**：AI 根据已有对话历史，判断当前条目的信息是否充足：

- 若用户回答已能清晰映射评分锚点 → **直接推进下一条目**
- 若信息不足或有值得深挖的细节 → **发起追问**

**追问铁律**：

- 每个条目适当追问，保证完成整个评估时间控制在 15 分钟以内
- 若几次追问仍信息不足，则触发兜底机制
- 追问需同时完成：自然承接用户对话 + 顺带锚定一个评分细节
- **追问后再追问——严禁**

**各条目追问方向参考**：


| 条目       | 追问方向                                     |
| -------- | ---------------------------------------- |
| Q1 兴趣    | 是什么事 / 兴趣减退的范围有多广                        |
| Q2 情绪    | 低落的深度 / 有没有绝望感                           |
| Q3 睡眠    | 睡多还是睡少（方向先于频率）                           |
| Q4 精力    | 程度 / 对日常生活的影响                            |
| Q5 食欲    | 吃多还是吃少（方向先于频率）                           |
| Q6 自我评价  | 具体怎么觉得自己不好 / 有没有觉得是负担                    |
| Q7 注意力   | 影响到哪些具体事情（学习/日常）                         |
| Q8 行动/语速 | 自己感觉到还是别人注意到                             |
| Q9 自伤意念  | 被动想法（消失就好）还是主动念头（有计划）——**仅确认是否有意念，不多追问** |


**正确追问示例**：

- 用户说：「感觉什么都懒得动。」
- 追问：「这种状态，最近是某几天特别明显，还是几乎每天都这样？」

**错误追问示例（调查感太强）**：

- 「从什么时候开始的？大概一周有几天？」——连续两问，且没有共鸣，像填表而非对话。



### 4.5 条目覆盖与动态顺序

AI **不按固定顺序提问**，根据用户上一句话的内容自然引出下一个待覆盖条目。系统后台维护条目覆盖状态追踪器。


| 机制      | 说明                                                      |
| ------- | ------------------------------------------------------- |
| 条目覆盖追踪  | 后台记录每个条目的状态：pending（待覆盖）/ answered（已覆盖）/ fallback（兜底作答） |
| 自然引出    | AI 从用户回答中找到与下一个待覆盖条目相关的线索，顺势引入                          |
| 兜底推进    | 若某条目无法自然引出，AI 用中性语气直接问，不强求叙事连贯                          |
| Q9 顺序约束 | Q9 不在前三轮触发，需在一定信任感建立后以改写方式引出                            |




### 4.6 兜底评分机制

当 AI 无法从自然对话中获取足够信息进行准确评分时，触发兜底机制。兜底分两个阶段进行——**先软性提示、再硬性选项**。

#### 第一阶段：软性提示

在触发硬性四选一兜底之前，AI 首先进行一次软性提示——以自然的对话方式，根据用户前面的模糊回答中已经透露的线索，试探性地提出一个最可能的匹配，以确认而非逼迫的方式征求用户的意见。

**软性提示的结构**（AI 动态生成，非固定话术）：

1. **承接与共情**：先接纳用户的模糊表达，不施加压力
2. **试探性匹配**：根据上下文中已有的线索，提出一个最可能的等级
3. **给予主动权**：邀请用户确认或修正，保持对话的开放性

**软性提示示例**（以 Q4 精力为例）：

> 学生（多轮模糊回答后）：「就是有时候觉得累，但也说不上多严重。」
>
> AI：「没关系，要准确说清楚累的程度确实不太容易。不过从你刚才聊的感觉来看——这种累，更像是有几天会比平时需要多一点力气才能做事，但大体上还是能完成的。你觉得这个感觉对吗？还是不太一样？」

**软性提示的结果分流**：


| 学生反应                   | 处理方式                                           |
| ---------------------- | ---------------------------------------------- |
| 确认（「嗯，差不多」「是的」等）       | 按照试探的等级直接记为该条目得分，不再触发硬性兜底                      |
| 修正（「其实比那个严重一点」「没那么多」等） | AI 根据学生的修正表达调整判断；若修正后信息充足则直接记分推进；若仍不够充分→进入硬性兜底 |
| 仍然不确定（「我也说不好」「不确定」等）   | 进入硬性兜底选项（第二阶段）                                 |




#### 软性提示的把握度分级（V3.3 新增）


| 把握度       | 判断依据                               | 软性提示处理方式                       |
| --------- | ---------------------------------- | ------------------------------ |
| **高（≥3）** | 学生虽模糊但提到了 ≥1 个明确线索：如具体频率词、时长、影响描述等 | 正常软性提示，用较确定的语气                 |
| **中（2）**  | 有部分线索但可多义解释，或在两极间摇摆                | 软性提示，语气更试探，强调确认而非默认            |
| **低（≤1）** | 几乎没有有效信号。学生回答极短，上下文无法支撑任何方向的推测     | **跳过软性提示，直接进入硬性兜底选项（第二阶段四选一）** |


**连续低把握跳过规则**：
同一个对话 Session 内，如果**连续 2 个条目**均因低把握（≤1）跳过了软性提示，说明该学生的表达风格整体偏模糊——从第 3 个条目起，**全部跳过软性提示直接走硬性兜底选项**，避免反复试探浪费时间。

#### 第二阶段：硬性兜底选项

不直接展示 0–3 分数值，而是将评分锚点转化为描述性语言，以四选一的方式呈现。

**交互规则**：

- 四个选项以按钮形式展示，用户必须选择其中一个才能继续对话
- 选项对应 0–3 分，用户选择后直接映射为该条目得分，不再追问
- AI 自然承接用户的选择，继续推进下一个条目，不强调「你刚才选了 X 分」
- 兜底选项的描述语言根据每个条目的锚点动态生成，不使用通用模板
- 给教师报告呈现的判断理由注明「用户通过兜底选项作答」



### 4.7 高风险条目（Q9）特殊处理

Q9（自杀/自伤意念）是所有条目中风险等级最高的，需要特殊处理：

- 不在前三轮触发，需在一定信任感建立后引出
- 以改写后的间接方式引出，不使用「不想活」「自杀」等词汇
- **主问题示例**：「有时候人在特别难受的时候会有一些不好的想法，比如希望一切都消失，或者想要伤害自己。你有没有这样的时候？」
- 用户说没有 → **立即接受**，不追问
- 用户说有，或模糊回答 → 需综合分析用户意图，确认回答真实度，若无法确认，则触发兜底机制
- 任何非零得分或兜底选项选择非零 → 立即标注高风险，触发后续安全流程



### 4.8 对话收尾（分级控制）

所有条目覆盖完毕、对话感觉自然收束时，AI 输出收尾语句。V3.2 将收尾从统一话术升级为分级策略：


| 风险等级      | 收尾策略    | 结束话术基调                      |
| --------- | ------- | --------------------------- |
| 低风险（0-4）  | 自然收束    | 「好的，我们聊完了。稍等一下，我来整理一下这次对话。」 |
| 轻度（5-9）   | 温和提醒    | 在自然收束基础上增加一句温和的关切表达         |
| 中度（10-14） | 表达关切    | 在收束中传递「你的感受值得被认真对待」的信号      |
| 高度（15-27） | 强调支持与安全 | 收束中强调支持和安全，语气稳定清晰           |


**收尾禁止行为**：

- 突然轻松化（如「好啦，没事的！」）
- 直接机械结束对话
- 产生突兀的情绪转换



### 4.9 对话中严格禁止


| 类别   | 禁止行为                                       |
| ---- | ------------------------------------------ |
| 结构暴露 | 暴露条目编号、量表名称、评分逻辑、条目总数                      |
| 问卷感  | 逐条提问、让学生感觉「在做问卷」、连续追问两个问题                  |
| 治疗干预 | 安慰（「一切都会好的」）、解释（「这是因为……」）、建议（「你应该……」）、过度共情 |
| 评估暴露 | 对话中给出任何评分或风险判断（「看来你的情况比较严重」等）              |
| 收束语  | 待覆盖条目不为空时说「聊到这」「今天我们聊了」「差不多了」「总的来说」等总结性话语  |
| 空洞共情 | 「我理解你」「这很正常」「很多人都这样」                       |

## 5. 评分引擎



### 5.1 评分方式：语义聚合 → 锚点匹配

**Step 1 · 语义聚合**：系统将所有轮次（主问题 + 追问）的用户回答合并为完整语义片段。若有兜底选项选择，直接使用选项对应分值，不参与语义判断。

**Step 2 · 锚点匹配**：将聚合后的语义与量表各等级的锚点描述（0-3 分）进行匹配，输出分数与判断理由。

> 学术基础：基于 "Evaluating the Efficacy of AI-Based Interactive Assessments Using Large Language Models for Depression Screening" 和 "ConvScale: Conversational Interviews for Scale-Aligned Measurement" 的 proven approach。



### 5.2 评分决策逻辑（逐条目执行）

对每个非兜底条目，按以下四步顺序判断：

1. **Step 1**：用户是否表现出任何该维度的异常？→ 否：0 分，结束；→ 是：继续
2. **Step 2**：症状是否达到最严重程度（3 分锚点描述）？→ 是：3 分，结束；→ 否：继续
3. **Step 3**：症状更接近 1 分还是 2 分描述？→ 给出 1 或 2 分
4. **Step 4**：输出分数 + 判断理由（内部使用，不展示给用户）

**评分参考维度**（不只频率，综合判断）：


| 维度     | 说明                         |
| ------ | -------------------------- |
| 频率     | 多少天出现、是否持续、偶发还是规律          |
| 强度     | 用户描述的程度，如「有点」vs「非常」vs「完全」  |
| 影响范围   | 是否影响到学习、社交、日常生活功能          |
| 自我觉察程度 | 主动描述越详细，往往反映症状越有意识地存在      |
| 语气与语境  | 犹豫、否认、轻描淡写可能低估；直接、具体的描述更可信 |

## 6. 报告引擎



### 6.1 设计哲学

报告的设计目标是让用户感受到：**被看见、被理解、被赋能**——看见自己已有的力量，从中获得继续向前走的信心。

- **不展示分数和等级标签**。学生版报告不使用「你的得分」「你属于 XX 级别」等表述。
- **去病理化**。不使用「症状」「抑郁」「障碍」「风险」等词汇。
- **锚定对话原文**。每一条观察和解释必须有对话中的具体内容作为依据；信息不足时使用兜底表述。
- **从分数到洞察**。系统不告诉学生「你得了多少分」，而是帮助他发现：自己已经拥有了哪些应对生活的力量。



### 6.2 学生版报告结构

学生报告由四个模块构成，严格按顺序呈现：

```
┌─────────────────────────────────────┐
│  ① 状态凝练句                        │
│  ← 10-15 字，从用户原话提炼，大字居中   │
├─────────────────────────────────────┤
│  ② 此刻的你                          │
│  第一层 · 观察（1 句）                 │
│  第二层 · 解释（3-4 句）               │
│  第三层 · 资源（2 句）                 │
├─────────────────────────────────────┤
│  ③ 往前一步                          │
│  引导语 + 建议 3-5 条                  │
│  [Q9 非零时：危机支持提示]             │
├─────────────────────────────────────┤
│  ④ 底部：重新开始                      │
└─────────────────────────────────────┘
```



#### 模块一：状态凝练句

报告顶部展示一句由 AI 从用户对话中凝练出的话，作为整份报告最显眼的视觉元素。凝练句不采用单次生成，而是走**「两选一 + 自检」**流程以保证输出质量：

1. **LLM 同时生成两个候选凝练句**，从不同角度切入（一个偏「承认处境，但不被处境定义」，一个偏「锚定用户身上已有的力量」）
2. **LLM 对每个候选做二元自检**——「这个句子是否满足：来自用户原话、落点在力量或韧性、不诗意不鸡汤、不引入用户没说过的概念？」
3. **择优展示**：两个都通过→随机选用一个；只有一个通过→展示那个；两个都不通过→使用固定兜底模板


| 属性  | 要求                             |
| --- | ------------------------------ |
| 来源  | LLM 从用户对话中提炼，所有候选必须锚定用户原话      |
| 长度  | 10–15 字，一句话，不折行                |
| 语气  | 保留用户原有的措辞风格和语气                 |
| 视角  | 描述性而非诊断性——说的是用户的感受和处境，不是对用户的评判 |
| 落点  | 落在用户身上存在的力量、清醒或韧性——承认困难，但有出路感  |
| 禁止  | 病理化词汇、散文化或诗意化表达                |


**兜底模板**（两个候选均未通过自检时使用）：

> 「完成了一次认真的对话，对自己的了解又多了一点。」



#### 模块二：此刻的你（三层递进）


| 层次       | 定位               | 篇幅    | 要求                |
| -------- | ---------------- | ----- | ----------------- |
| 第一层 · 观察 | 说出用户身上正在发生什么     | 1 句   | 不列举清单，不说教         |
| 第二层 · 解释 | 说明这些感受和状态背后的心理逻辑 | 3-4 句 | 日常语言，不推断童年经历/家庭模式 |
| 第三层 · 资源 | 发现用户已经在发挥的力量     | 2 句   | 来自真实对话内容，用事实说话    |




#### 模块三：往前一步（差异化建议）

按「发现你的状态→理解你的需要→匹配你的方向」逻辑展开，每条 30-40 字。

## 7. 安全与风险管理

安全机制是 COSMO 最重要的非功能需求。系统从对话、评分、报告三个阶段对风险进行层层拦截。

### 7.1 三层安全防护

```
第一层 · 对话阶段 ──→ 高风险实时检测 ──→ 意图确认 ──→ 固定确认界面 ──→ 兜底引出 Q9 ──→ 情绪稳定后继续
第二层 · 评分阶段 ──→ Q9 强制规则 ──→ 风险升级（severe）
第三层 · 报告阶段 ──→ 危机热线固定模块 ──→ 不可绕过
```



### 7.2 第一层：对话阶段高风险实时检测

**检测范围**：在对话过程中实时识别用户消息中的以下风险语义：

- 自伤倾向（如「我想伤害自己」等表达）
- 自杀表达（如「不想活了」「死了算了」等）
- 极端绝望（如「没有意义」「一切都完了」「不想存在」等）

**响应流程（五步递进）**：

#### Step 1 · 意图确认

系统检测到高风险语义后，AI 不立即触发兜底机制，而是先通过自然对话确认用户意图的真实性和严重程度。AI 需从以下两个维度进行判断：

- **情绪强度**：用户表达的痛苦程度如何？是短暂的、情境性的表达，还是持续存在的、弥漫性的感受？
- **行为程度**：是否有具体的行为迹象？（如「我已经想好了」「我有办法」vs「有时候会有这种感觉」）

确认过程中 AI 的回应需同时满足三个要素：

1. **情绪承接**——从用户的情绪强度和具体处境出发，做出有针对性的回应，而非使用通用话术
2. **共情回应**——不评判、不淡化、不急于转向积极。承认痛苦的真实性
3. **支持性表达**——自然引出陪伴和资源的提示，不突兀、不说教

**意图确认示例**：

> 学生：「有时候真的觉得活着没什么意思。」
>
> AI：「听你这么说，我能感受到你现在心里很重——这种觉得没意思的感觉，是偶尔一闪而过，还是最近一直在心里转？」



#### Step 2 · 判断与分流


| 判断结果                        | 处理方式                 |
| --------------------------- | -------------------- |
| 非真实高风险意图（情境性表达、无主动意念、无行为迹象） | 继续当前对话流程，保持对后续表达的敏感性 |
| 确认真实高风险意图（有被动或主动意念）         | 进入 Step 3 触发 Q9 兜底机制 |




#### Step 3 · 触发 Q9 兜底机制

确认高风险意图真实存在后，将当前条目标记为 Q9，并触发该条目的兜底评分机制——将评分锚点转化为描述性语言，以四选一的方式呈现：

> 「谢谢你愿意和我说这些。为了更好地了解你现在的状态，你可以选一个最接近的——
>
> ① 偶尔会有这种一闪而过的念头，但不会真的去想
> ② 有过这样的想法，但没有想过具体要怎么做
> ③ 认真想过具体怎么做，或者什么时候做
> ④ 已经有具体的计划，或者曾经尝试过」

用户选择后，选项直接映射为 Q9 得分（0-3 分），不再追问任何细节。

#### Step 4 · 情绪稳定后继续

Q9 兜底选项完成后，AI 不立即推进到下一个条目，而是与用户进行 **1-3 轮**的情感承接对话：

承接对话原则：

- 语言自然过渡，不产生突兀的情绪转换
- 以肯定用户的坦诚和勇气为基调（「谢谢你愿意和我说这些」）
- 表达持续的支持态度（「接下来的对话，如果你觉得哪里不舒服，随时可以和我说」）
- 给用户一个轻的落点，让情绪自然回落到可以继续对话的状态

情绪承接对话结束后，系统展示一个短的固定确认问题（不由 LLM 生成）：

> 「你现在感觉可以继续聊其他话题了吗？」

两个固定按钮：「可以继续」/「先休息一下」

学生选择「先休息一下」→ 展示安全提示，同时保留继续入口。
学生选择「可以继续」→ AI 自然推进到下一个待覆盖条目，继续完成其余条目的对话。

### 7.3 第二层：评分阶段 Q9 强制规则

- Q9 任何非零得分 → risk_level 强制设为 severe
- q9_nonzero 标记为 true，不受其他条目得分影响
- 此规则**硬编码于评分逻辑中**，不依赖模型输出，无法被绕过



### 7.4 第三层：报告阶段安全模块


| 对象     | 安全模块内容                             |
| ------ | ---------------------------------- |
| 学生报告   | 建议模块最前端插入固定危机热线文案，不由 AI 生成，不可被省略   |
| 教师个案报告 | 顶部醒目位置标注高风险：有自杀/自伤意念，附带对话摘录，标明意念性质 |
| 教师汇总报告 | 高风险学生数量单独列出，触发班主任个案跟进提醒            |




### 7.5 危机热线与资源

- 全国心理援助热线：**400-161-9995**（24 小时）
- 希望 24 热线：400-161-9995
- 校内心理老师联系方式（由学校管理员配置）

## 8. 引擎可复用性设计

COSMO 引擎与量表解耦。PHQ-9 是当前的示范量表——换量表只需替换配置，引擎核心不变。

### 8.1 量表配置文件

每个量表对应一个独立配置文件（`src/lib/scales/*.ts`），包含：

- 量表名称和描述（用于开场透明告知文案）
- 条目列表：原始条目文本 + core_intent + 追问方向 + is_high_risk 标记 + fallback_question + fallback_options（四个，对应 0-3 分）+ score_anchors（四个，对应 0-3 分的语义锚点）
- 风险分级阈值（minimal / mild / moderate / severe 的分值范围）
- 高风险条目的报告融入话术（如适用）



### 8.2 引擎不变的部分（换量表无需改动）


| 模块   | 不变的内容                                |
| ---- | ------------------------------------ |
| 开场引擎 | 透明告知框架、破冰对话逻辑                        |
| 对话引擎 | 单次回应结构（共鸣·追问决策）、条目覆盖追踪机制、快捷短语、兜底评分机制 |
| 评分引擎 | 四步评分决策逻辑（Step 1-4）、评分参考维度            |
| 报告引擎 | 学生报告三层结构（凝练句·状态分析·有方向）、教师报告结构        |
| 安全系统 | 三层安全防护、高风险实时检测、Q9 强制规则               |
| UI 层 | 对话页结构、报告页布局、动效规范                     |




### 8.3 接入新量表所需改动

- 新增量表配置文件（条目原文、core_intent、追问方向、高风险标记、fallback 配置）
- 更新开场透明告知中的量表名称和描述文字
- 更新高风险条目的报告融入话术（如适用）
- 调整风险分级阈值（如需）

UI、对话逻辑框架、报告结构、System Prompt 主体均无需改动。

## 9. UI 设计系统



### 9.1 设计方向：Light Mode 校园访谈氛围

V3.2 将视觉体系全面升级为 Light Mode（校园访谈氛围）。

**设计目标**：

- **安全感**——色调温暖、不压迫，让学生感到安心而非紧张
- **去病理化**——不传递任何「医院」「诊所」「测试」的视觉暗示
- **非审判**——克制、不喧哗，像一间安静谈话的教室
- **校园环境适配**——明亮但不刺眼，适合日常教学环境中使用



### 9.2 配色规范


| 角色     | 色值             | 用途               |
| ------ | -------------- | ---------------- |
| 背景色    | #F7F6F2（米白）    | 全局页面背景，温和不刺眼     |
| 主色     | #A3B5A0（浅鼠尾草绿） | 按钮、强调元素、进度指示     |
| 文字主色   | #2D2D2D（深灰）    | 正文、标题            |
| 文字辅色   | #787878（中灰）    | 说明文字、占位文本        |
| AI 气泡  | #F2F0ED（米灰）    | 接近访谈记录纸的质感，左对齐   |
| 用户气泡   | #E8EEF2（淡蓝灰）   | 与 AI 气泡有明显区分，右对齐 |
| 分隔线/边框 | #E5E2DE        | 卡片边框、分隔线         |
| 积极指示   | #7BAE7F        | 低风险标签、正面反馈       |

### 9.3 页面设计要点



#### 开始页

- 居中布局，克制文字。产品名 + 一行说明 + 开始按钮
- 不使用 Hero 大标题或装饰元素
- 底部小字：隐私说明
- 三个小图标（安全保密/自然对话/了解自己）+ 短标签，建立信任感



#### 对话页


| 元素    | 设计要点                                            |
| ----- | ----------------------------------------------- |
| 聊天气泡  | 轻量、接近访谈记录纸质感。AI 气泡左对齐（#F2F0ED），用户气泡右对齐（#E8EEF2） |
| 进度指示  | 极简：只显示大致阶段（「开始中」「快结束了」），不暴露条目数和编号               |
| 输入区   | 简洁：输入框（圆角 16px）+ 发送按钮 + 快捷短语小按钮。固定于页面底部         |
| 快捷短语  | 输入框旁小按钮，用户主动点击后展开 chip 列表，不自动弹出                 |
| 空状态   | 页面加载后 AI 自动发出第一条消息，无需用户等待                       |
| 打字指示器 | 三个小圆点依次弹跳                                       |




#### 承接页

- 居中布局，一句过渡语 +「查看报告」按钮
- 用户主动点击进入报告页，不自动跳转
- 极简动画，慢而有呼吸感



#### 学生报告页


| 元素   | 设计要点                     |
| ---- | ------------------------ |
| 凝练句  | 10-15 字大字居中显示，充足留白，截图友好  |
| 分析段落 | 行距宽松，段间留白明显              |
| 建议列表 | 每条独立呈现为卡片或分隔段落           |
| 危机提示 | 融合在建议开头，柔和暖色背景轻微区分，不用警告红 |




### 9.4 动效原则

- 所有过渡使用 ease-out 曲线，时长 300–500ms
- 气泡以轻微上浮 + 淡入方式出现（y: 10px, opacity: 0 → 1）
- 只做必要的功能性动效，不做装饰性动效
- 动效节奏宁慢不偏快，保持沉静感
- 不做快速弹出、跳切、缩放等刺激感动效



### 9.5 响应式要求

- 学生端：优先适配手机（375-414px 宽度）
- 教师端：适配桌面（1024px+）
- 聊天气泡在手机上最大宽度 ≤ 80%
- 文本不随视口缩放，保持 15px 基准字号

## 10. 技术架构



### 10.1 技术选型


| 模块       | 技术方案                        | 说明                         |
| -------- | --------------------------- | -------------------------- |
| 框架       | Next.js 14（App Router）      | 服务端渲染 + API Routes，统一前后端   |
| 语言       | TypeScript                  | 全栈类型安全                     |
| 样式       | Tailwind CSS + 自定义 CSS      | Utility-first + 柔化渐变/模糊/动效 |
| 字体       | PingFang SC / Noto Sans SC  | 中文优先，屏幕阅读体验好               |
| LLM 服务   | DeepSeek API（deepseek-chat） | OpenAI SDK 兼容格式，服务端调用      |
| LLM 调用位置 | 服务端 API Routes              | API Key 不暴露给前端             |
| 量表配置     | `src/lib/scales/*.ts`       | 每个量表独立配置文件，与引擎解耦           |
| 部署       | Vercel / 自托管                | 支持 Edge Functions          |
| API Key  | 环境变量 `DEEPSEEK_API_KEY`     | 服务端持有，前端不可见                |




### 10.2 API Route 设计


| Route              | 功能     | 输入                         | 输出                             | 关键逻辑              |
| ------------------ | ------ | -------------------------- | ------------------------------ | ----------------- |
| `POST /api/chat`   | 对话主接口  | messages[] + session_state | AI 回复 + 更新后的 session_state     | 条目覆盖追踪、高风险检测      |
| `POST /api/score`  | 评分接口   | 完整对话记录 + fallback_scores   | 各条目分数 + 总分 + risk_level + 判断理由 | 语义聚合→锚点匹配→Q9 强制规则 |
| `POST /api/report` | 报告生成接口 | 评分结果 + 对话摘要                | 凝练句 + 状态分析 + 建议列表              | 三层报告结构分段生成        |




### 10.3 数据流

```
用户消息 → POST /api/chat
  ├─ 读取 session_state（条目覆盖状态）
  ├─ 组装 System Prompt + 对话历史
  ├─ DeepSeek API → AI 回复
  ├─ 高风险语义检测
  └─ 返回 { reply, session_state }

对话结束 → POST /api/score
  ├─ 提取完整对话文本
  ├─ 组装评分 System Prompt（含锚点 + fallback_scores）
  ├─ DeepSeek API → 评分 JSON
  ├─ Q9 强制规则覆盖
  └─ 返回 { scores, total, risk_level, q9_nonzero }

评分完成 → POST /api/report
  ├─ 组装报告 System Prompt（含评分结果）
  ├─ DeepSeek API → 报告 JSON
  └─ 返回 { condensed_sentence, status_analysis, suggestions }
```



### 10.4 Session 状态管理

Demo 阶段使用前端 React Context / 服务端内存 Map 持有状态（对话历史、条目覆盖状态、评分结果、报告数据）。正式产品化后迁移至服务端 Session 或数据库。

## 12. 附录



### A. PHQ-9 九条目完整配置

各条目包含完整的原始条目文本、核心测量意图、追问方向、高风险标记、兜底选项（fallback_question + fallback_options[0-3]）、评分锚点（score_anchors[0-3]）。详见 §5.3 及代码仓库 `src/lib/scales/phq9.ts`。

### B. 关键学术参考

- **"Evaluating the Efficacy of AI-Based Interactive Assessments Using Large Language Models for Depression Screening: Development and Usability Study"** — 语义聚合→锚点匹配→分数，奠定了本产品评分引擎的核心方法论
- **"ConvScale: Conversational Interviews for Scale-Aligned Measurement"** — 对话片段→提取证据陈述→对照锚点打分 + 理由



### C. 风险等级去病理化表述对照


| 专业等级            | 学生报告用词 | 含义                        |
| --------------- | ------ | ------------------------- |
| minimal（0-4）    | 状态良好   | 情绪状态良好，目前没有需要特别关注的信号      |
| mild（5-9）       | 稍有起伏   | 最近情绪有些起伏，可能需要更多关注自己的状态    |
| moderate（10-14） | 需要关注   | 你在承受比较大的心理压力，建议认真对待并寻求支持  |
| severe（15-27）   | 需要支持   | 你目前的状态需要专业的支持和帮助，请不要一个人扛着 |




### D. 术语表


| 术语          | 英文/缩写                          | 定义                                             |
| ----------- | ------------------------------ | ---------------------------------------------- |
| PHQ-9       | Patient Health Questionnaire-9 | 九条目患者健康问卷，用于抑郁症状筛查                             |
| 构念一致性       | Construct Validity             | AI 改写后的提问是否真实覆盖了量表条目原本要测量的心理学构念                |
| 锚点          | Anchors                        | 每个分值对应的语义描述，是评分匹配的参照标准                         |
| 语义聚合        | Semantic Aggregation           | 将多轮分散的用户回答整合为完整的待评分语义片段                        |
| 兜底机制        | Fallback                       | 当自然对话无法获取足够评分信息时，用四选一按钮收集所需信息                  |
| QLV         | Question Validation Layer      | 问题校验层，校验 AI 提问是否有效覆盖核心意图                       |
| core_intent | 核心意图                           | 量表条目的核心测量目标，AI 提问不得偏离或改变此含义                    |
| risk_level  | 风险等级                           | minimal / mild / moderate / severe 四级，决定报告建议策略 |
